## 1. Резюме проекта

Проект SMM Plan представляет собой полнофункциональную SMM-панель, построенную на стеке **Next.js 15 + Prisma ORM + PostgreSQL + Redis**. Система обеспечивает продажу услуг по продвижению в социальных сетях (подписчики, лайки, просмотры) через веб-сайт, Telegram Mini App и Telegram-бота. Проект включает модули каталогизации услуг, обработки заказов, интеграции с платёжными системами (ЮKassa, Робокасса), реферальную программу, систему лояльности, предиктивный анализ оттока (churn prediction), административную панель и AI-модули для генерации контента.

В рамках комплексного аудита были проведены три параллельных направления анализа: аудит безопасности (уязвимости и угрозы), аудит бизнес-логики (логические ошибки и архитектурные проблемы) и маркетинговый аудит (стратегия, UX, SEO, конверсия).

Маркетинговая стратегия проработана на высоком уровне, но имеет критичные пробелы в реализации (фейковые отзывы, отсутствие бесплатного тест-драйва, незавершённый pSEO).

### 2.1. Критические уязвимости (CRITICAL)

#### V-01: Захардкоженный Telegram Bot Token в исходном коде

- **Файл:** `src/scripts/create-test-project.ts:11`
- **Код:** `const token = '8580696545:AAEOttbpfVTmGyQMNHQ5e22WBKpKiBq6zVc';`
- **Описание:** Реальный токен Telegram бота вшит прямо в исходный код. Токен даёт полный контроль над ботом — чтение сообщений, отправку от имени бота, управление вебхуками.
- **Влияние:** Полная компрометация Telegram бота. Атакующий может перехватывать сообщения пользователей, рассылать спам, управлять вебхуками.
- **Рекомендация:** Немедленно отозвать токен через @BotFather. Удалить из кода. Перенести в переменные окружения. Добавить pre-commit хук на поиск токенов.

#### V-02: Захардкоженный пароль администратора в исходном коде

- **Файл:** `scripts/create-first-admin.ts:30`
- **Код:** `const hashedPassword = await bcrypt.hash('admin12345678', 10);`
- **Описание:** Пароль супер-администратора `admin12345678` захардкожен в скрипте инициализации. Скрипт также выводит пароль в консоль.
- **Влияние:** Если этот скрипт был запущен на продакшене, пароль даёт полный доступ к админ-панели с правами Super Admin.
- **Рекомендация:** Немедленно сменить пароль администратора. Изменить скрипт на генерацию случайного пароля или чтение из env. Никогда не выводить пароли в лог.

#### V-03: Дампы БД с реальными данными пользователей в репозитории

- **Файл:** `backups/local_smmplan_backup_2026-03-17_23-41-37.sql` и другие (9 файлов, ~3.5MB)
- **Описание:** В директории `backups/` хранятся SQL-дампы с реальными пользовательскими данными: хеши паролей, email, Telegram ID, 2FA-коды, API-ключи, балансы, токены ботов проектов. Файлы НЕ исключены из `.gitignore`.
- **Подтверждённые данные в дампе:** `COPY public."User" (...password, twoFactorCode, apiKey, email...)`
- **Влияние:** Компрометация персональных данных всех пользователей. Утечка финансовых данных. Нарушение GDPR/ФЗ-152. Хеши паролей могут быть подобраны. 2FA-коды хранятся в открытом виде.
- **Рекомендация:** Немедленно удалить все бэкапы из репозитория и из Git-истории (`git filter-branch`). Добавить `backups/` в `.gitignore`. Переместить бэкапы в защищённое хранилище. Сбросить все 2FA-коды. Уведомить пользователей о смене паролей.

#### V-04: Отсутствие middleware.ts — нет централизованной защиты админ-маршрутов

- **Файл:** Отсутствует `src/middleware.ts` или `middleware.ts`
- **Описание:** В проекте НЕТ Next.js middleware. Все маршруты `/api/admin/*` не защищены на уровне middleware — защита реализована только внутри каждого обработчика route.ts индивидуально. Если разработчик забудет добавить проверку сессии в новом админском эндпоинте, он будет доступен без авторизации.
- **Влияние:** Любой новый админский API-эндпоинт без явной проверки сессии становится публично доступным.
- **Рекомендация:** Создать `src/middleware.ts` с проверкой `admin_session` cookie для всех маршрутов `/api/admin/*` и `/admin/*`.

## 3. Аудит бизнес-логики

Аудит бизнес-логики выявил 22 проблемы, из которых 4 являются критическими и ведут к прямым финансовым потерям. Основная архитектурная проблема — дублирование путей создания заказов, при котором ключевые проверки безопасности и биллинга обходятся в альтернативных API-маршрутах.

### 3.1. Критические ошибки бизнес-логики

#### L-01: Дублирование путей создания заказов — обход B2B-биллинга

- **Файлы:** `src/app/api/tma/orders/route.ts`, `src/app/api/client/orders/route.ts`, `src/services/orders/order-activation.service.ts`
- **Описание:** В системе существует 3 независимых пути создания заказов:
  1. **`OrderActivationService.initiateOrder()`** — содержит B2B-биллинг, PricingGuard, promo-трекинг, loyalty-проверки
  2. **TMA API** — создаёт заказ напрямую через `prisma.$transaction()`, **минуя** `OrderActivationService`. Нет B2B-биллинга, нет PricingGuard
  3. **Web API** — аналогично, создаёт заказы напрямую для авторизованных пользователей
- **Влияние:** B2B-клиенты получают услуги бесплатно; PricingGuard не вызывается; costPrice рассчитывается некорректно
- **Рекомендация:** Унифицировать все пути создания заказов через `OrderActivationService.initiateOrder()`

#### L-02: YooKassa — отсутствие истинной идемпотентности платежей

- **Файл:** `src/services/payments/yookassa.service.ts:29`
- **Описание:** Idempotency-Key генерируется как `crypto.randomUUID()` при **каждом** вызове `createPayment()`. Если клиент нажмёт «Оплатить» дважды, будет создан **второй** платёж с новым ключом.
- **Влияние:** Двойное списание средств или двойное зачисление на баланс
- **Рекомендация:** Генерировать Idempotency-Key детерминистически на основе `transaction.id`

#### L-03: Drip Feed — потеря единиц при целочисленном делении

- **Файл:** `src/services/orders/drip-feed.service.ts:82`
- **Код:** `const qtyToOrder = Math.floor(order.quantity / order.runs);`
- **Описание:** При `quantity=1000, runs=3`: каждый run = `Math.floor(1000/3)` = 333. Итого: 333 × 3 = 999. Одна единица потеряна.
- **Влияние:** Клиент платит за 1000 единиц, но получает 999. Систематическая потеря на каждый drip-feed заказ.
- **Рекомендация:** Последний run должен получать остаток: `(runIndex < runs - 1) ? Math.floor(qty/runs) : qty - Math.floor(qty/runs) * (runs - 1)`

#### L-04: Robokassa — зачисление суммы из БД вместо суммы из платежа

- **Файл:** `src/app/api/webhooks/robokassa/route.ts:110`
- **Описание:** Вебхук credит баланс на `currentTx.amount` (из БД), но **не проверяет**, что `outSum` (реально оплаченная сумма) совпадает.
- **Влияние:** Некорректное зачисление при расхождении сумм
- **Рекомендация:** Добавить проверку `Math.abs(parseFloat(outSum) - currentTx.amount.toNumber()) > 0.01`

## 5. Рекомендуемые узкоспециализированные аудиты

На основании проведённого комплексного анализа были выявлены области, требующие углублённого изучения специалистами соответствующего профиля. Ниже представлен полный список из 28 рекомендуемых аудитов.

### 5.1. Аудиты безопасности

#### SA-01: Penetration Testing (Pentest) — 🔴 CRITICAL

Полноценный пентест веб-приложения и API по OWASP Top 10. Включает тестирование на SQL-инъекции, XSS, CSRF, IDOR, broken access control, security misconfiguration. Особый фокус на админ-панель и платёжные эндпоинты.

**Обоснование:** 4 CRITICAL и 9 HIGH уязвимости в первичном аудите требуют подтверждения эксплуатируемости.

#### SA-02: Аудит утечки данных (Data Leak Audit) — 🔴 CRITICAL

Анализ Git-истории на предмет утечек секретов, ключей, паролей. Сканирование всех коммитов через git-secrets, truffleHog, gitleaks. Проверка бэкапов БД на наличие реальных пользовательских данных. Оценка соответствия ФЗ-152 и GDPR.

**Обоснование:** Бэкапы БД с хешами паролей и 2FA-кодами в открытом репозитории.

#### SA-03: Аудит платёжной безопасности (PCI DSS Compliance) — 🟠 HIGH

Проверка соответствия требованиям PCI DSS для обработки платежей. Анализ хранения карточных данных, безопасности вебхуков, идемпотентности транзакций, защиты от двойного списания.

**Обоснование:** Проблемы с идемпотентностью YooKassa, зачисление сумм без верификации.

#### SA-04: Аудит аутентификации и авторизации — 🟠 HIGH

Глубокий анализ всех потоков аутентификации: Telegram Widget Auth, TMA Auth, Admin Auth, NextAuth. Тестирование на privilege escalation, session hijacking, timing-атаки. Проверка ротации JWT-токенов.

**Обоснование:** Обход 2FA, хардкод мастер-ключа, timing-уязвимости.

#### SA-05: Аудит секретов и управления доступом (Secrets Management) — 🟠 HIGH

Инвентаризация всех секретов в проекте: API-ключи, токены, пароли, сертификаты. Внедрение HashiCorp Vault или AWS Secrets Manager. Настройка pre-commit хуков для предотвращения утечек. Ротация скомпрометированных ключей.

**Обоснование:** Захардкоженный токен бота, пароль администратора в коде.

#### SA-06: Аудит инфраструктурной безопасности — 🟡 MEDIUM

Анализ Docker-конфигурации, nginx-настроек, SSL/TLS сертификатов, сетевой изоляции контейнеров. Проверка docker-compose на утечки переменных окружения. Аудит Redis и PostgreSQL конфигурации.

**Обоснование:** Несколько Docker Compose файлов, публично доступные конфигурации.

#### SA-07: Аудит зависимостей (Supply Chain Security) — 🟡 MEDIUM

Сканирование `package-lock.json` через npm audit, Snyk или Dependabot. Анализ transitive-зависимостей. Проверка на известные CVE. Внедрение автоматического сканирования в CI/CD.

**Обоснование:** Проект не содержит результатов аудита зависимостей.

#### SA-08: Аудит логирования и мониторинга безопасности — 🟡 MEDIUM

Анализ полноты логирования security-событий: попытки входа, изменение привилегий, финансовые операции. Настройка Sentry для security-алертов. Удаление отладочных `console.log` с чувствительными данными.

**Обоснование:** `console.log` с email пользователей в продакшн-коде.

### 5.2. Аудиты бизнес-логики и финансов

#### BA-01: Аудит финансовой целостности (Financial Integrity Audit) — 🔴 CRITICAL

Полная сверка ledger с реальными балансами всех пользователей. Проверка каждой транзакции на корректность balanceBefore/balanceAfter. Анализ расхождений и их причин. Внедрение автоматического сверочного скрипта.

**Обоснование:** Ledger drift, отсутствие LedgerEntry для external refund.

#### BA-02: Аудит потоков создания заказов — 🔴 CRITICAL

Моделирование всех путей создания заказов (Web API, TMA API, Bot, Admin) с проверкой прохождения через каждый guard (PricingGuard, B2B-биллинг, loyalty, promo). Унификация через OrderActivationService.

**Обоснование:** 3 независимых пути, обходящих критические проверки.

#### BA-03: Аудит промо- и скидочной системы — 🟠 HIGH

Анализ всех комбинаций скидок (loyalty + early bird + promo) на предмет продажи ниже себестоимости. Добавление `MAX_TOTAL_DISCOUNT`. Проверка каждого промокода на наличие лимитов.

**Обоснование:** Стекинг скидок до 50-60%, промокоды без `maxUses`.

#### BA-04: Аудит реферальной системы — 🟠 HIGH

Анализ эксплуатируемости реферальной программы: self-referral, мульти-аккаунтинг, накрутка через мелкие депозиты. Внедрение антифрод-проверок (IP, device fingerprint, временные окна). Ограничение бонуса на первые N депозитов.

**Обоснование:** Бонус на каждый депозит, нет проверки self-referral.

#### BA-05: Аудит drip-feed и scheduled orders — 🟠 HIGH

Проверка корректности распределения единиц по run-ам. Тестирование edge cases (quantity=1, runs=100). Анализ обработки ошибок при пересчёте цен для scheduled orders.

**Обоснование:** Потеря единиц при целочисленном делении, использование устаревших цен.

#### BA-06: Аудит мультитенантной изоляции — 🟡 MEDIUM

Проверка всех API-эндпоинтов на фильтрацию по projectId. Анализ возможности доступа к данным другого проекта. Тестирование tenant resolution на подмену заголовков.

**Обоснование:** `getUserOrders` без projectId, tenant через X-Forwarded-For.

#### BA-07: Аудит платёжных вебхуков — 🟡 MEDIUM

Тестирование всех платёжных вебхуков на дублирование, подделку, race conditions. Проверка idempotency-key для всех платёжных провайдеров. Нагрузочное тестирование.

**Обоснование:** Случайный Idempotency-Key в YooKassa, отсутствие проверки сумм в Robokassa.

### 5.3. Маркетинговые и UX аудиты

#### MA-01: Аудит конверсионной воронки (CRO Audit) — 🔴 CRITICAL

Полный анализ воронки от первого посещения до повторной покупки. Настройка Google Analytics 4 / Yandex Metrika с событиями каждого шага. A/B тестирование ключевых элементов (HubInput, CTA, pricing). Карты кликов и scroll-глубина через Hotjar. Расчёт стоимости привлечения по каналам.

**Обоснование:** Отсутствие аналитики конверсии, фейковые отзывы подрывают доверие.

#### MA-02: Аудит SEO и pSEO (Technical SEO Audit) — 🟠 HIGH

Полный технический аудит: crawlability, indexability, Core Web Vitals, structured data, canonical URLs, robots.txt, sitemap.xml. Реализация pSEO-маршрута `/buy/[platform]/[action]/[modifier]`. Создание OG-изображений для каждой платформы. Исправление OrganizationJsonLd.

**Обоснование:** pSEO не реализован, битый logo в JSON-LD, нет robots.txt.

#### MA-03: Аудит юридического соответствия (Legal Compliance Audit) — 🔴 CRITICAL

Аудит на соответствие ФЗ-152 (персональные данные), ФЗ-54 (кассовый учёт), Правилам оказания услуг. Добавление чекбокса согласия на обработку ПД. Внедрение granular cookie consent. Проверка оферты, политики конфиденциальности, правил возврата.

**Обоснование:** Отсутствие согласия на ПД, нет granular cookies.

#### MA-04: Аудит контент-стратегии — 🟠 HIGH

Разработка контент-плана на 6 месяцев: 2-3 статьи в неделю, pSEO-лендинги, глоссарий 30+ терминов. Дописывание 6 статей-заглушек. Создание блога. Настройка email-рассылки.

**Обоснование:** 6 статей-заглушек, 8 терминов в глоссарии вместо 30+.

#### MA-05: Аудит пользовательского опыта (UX Audit) — 🟠 HIGH

Юзабилити-тестирование с 5-8 представителями целевой аудитории. Анализ тепловых карт и записей сессий. Устранение дублирования second-site/. Оптимизация анимаций для производительности. Упрощение InstantOrder wizard.

**Обоснование:** Два стиля компонентов, перегруженность анимациями.

#### MA-06: Аудит реферальной и адвокаси программы — 🟠 HIGH

Редизайн реферальной программы: кастомные коды, веб-страница, шеринг в соцсети, viral loop. NPS-опрос через web-интерфейс (не только Telegram). Визуальный прогресс реферала.

**Обоснование:** Реферальный код = ID пользователя, нет веб-рефералки.

#### MA-07: Аудит Telegram Mini App — 🟡 MEDIUM

Редизайн TMA как основного канала для B2B-аудитории. «Заказ в 2 клика» для повторных заказов. TMA-аналитика (events, session duration, conversion). Push-уведомления через Bot API.

**Обоснование:** TMA — обёртка над web-версией, а не нативный опыт.

#### MA-08: Аудит конкурентного позиционирования — 🟡 MEDIUM

Автоматический мониторинг цен и фич 5 основных конкурентов. Создание страницы «Почему мы». Comparison-лендинги для SEO. Квартальный обзор конкурентного ландшафта.

**Обоснование:** Анализ только primelike.ru, нет текущего мониторинга.

### 5.4. Аудиты качества кода и архитектуры

#### CA-01: Аудит архитектуры и технического долга — 🟠 HIGH

Полный анализ архитектурных решений: дублирование бизнес-логики, циклические зависимости, god-objects. Оценка технического долга в человеко-днях. План рефакторинга с приоритетами.

**Обоснование:** 3 независимых пути создания заказов, дублирование в second-site/.

#### CA-02: Аудит тестового покрытия — 🟠 HIGH

Измерение покрытия unit-тестами (цель: 80% для критических модулей). Анализ playwright e2e-тестов. Добавление интеграционных тестов для платёжных вебхуков.

**Обоснование:** 4 failed e2e-теста в test-results/, `ignoreBuildErrors` в next.config.js.

#### CA-03: Аудит производительности (Performance Audit) — 🟡 MEDIUM

Lighthouse-аудит: Core Web Vitals (LCP, FID, CLS). Анализ bundle-size. Lazy loading для framer-motion. Оптимизация запросов к БД (N+1 queries). Кэширование стратегия.

**Обоснование:** Перегруженность анимациями, churn batch без пагинации.

#### CA-04: Аудит доступности (Accessibility Audit) — 🟡 MEDIUM

Проверка на соответствие WCAG 2.1 AA: контрастность, keyboard navigation, ARIA-атрибуты, screen reader.

**Обоснование:** Проект не проходил accessibility-аудит, содержит сложные интерактивные компоненты.

#### CA-05: Аудит DevOps и CI/CD — 🟡 MEDIUM

Анализ pipeline: автоматическая сборка, тестирование, развёртывание. Настройка branch protection. Добавление pre-commit хуков (lint, type-check, secret scanning). Автоматический dependency update.

**Обоснование:** `ignoreBuildErrors` в продакшн-сборке, нет автоматического сканирования.
