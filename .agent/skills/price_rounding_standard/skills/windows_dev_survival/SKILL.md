# Навык: Выживание в Windows (PowerShell & CSP)

Этот файл содержит правила и рекомендации по избежанию распространенных ошибок при разработке проекта Smmplan в среде Windows.

## 1. Работа с PowerShell и Node.js (tsx/eval)

### Проблема: Символ `$` в командах
В PowerShell символ `$` зарезервирован для переменных. Команда `npx tsx -e "prisma.$disconnect()"` приведет к тому, что PowerShell удалит `$disconnect`, оставив `prisma.()`, что вызовет синтаксическую ошибку.

### Правило:
1. **Никогда не используйте сложные однострочники через `-e` в PowerShell**, если в них есть символ `$`.
2. **Лучший путь:** Создайте временный файл в папке `scripts/`, например `scripts/tmp-task.ts`, и запустите его:
   ```powershell
   npx tsx scripts/tmp-task.ts
   ```
3. Если нужно использовать `-e`, используйте одинарные кавычки для всей строки:
   ```powershell
   npx tsx -e 'console.log("Hello $world")'
   ```

## 2. Настройка CSP (Content Security Policy)

### Проблема: Блокировка внешних запросов
При добавлении новых API (Telegram, платежки, курсы валют) браузер блокирует запросы, если домен не внесен в `connect-src`.

### Правило:
1. При добавлении любой интеграции, использующей `fetch` или `iframe`, **обязательно** обновите `src/configs/security-headers.config.js`.
2. Проверьте:
   - `connect-src`: для API запросов.
   - `frame-src`: для вставки виджетов (например, Telegram Login).
   - `img-src`: если загружаются изображения с внешних хостов.

## 3. Использование инструментов правки кода (Replace/Multi-Replace)

### Проблема: "Target content not found"
Инструмент замены чувствителен к невидимым символам (пробелы, переносы строк `\r\n` vs `\n`).

### Правило:
1. Для замены выбирайте уникальный, но **короткий** фрагмент кода.
2. Не включайте в `TargetContent` слишком много пустых строк или комментариев.
3. Если замена не проходит, используйте `view_file`, чтобы скопировать текст в точности так, как его видит система.

## 4. Отладка внешних провайдеров

### Проблема: Непонятные ошибки от API (400, 500)
Провайдеры кидают ошибки типа `user_inactive`, которые легко спутать с багом в коде.

### Правило:
1. Всегда логируйте **полный ответ** от сервера провайдера при ошибках в `UniversalProvider` или `ProviderService`.
2. Перед отладкой кода проверьте статус аккаунта (баланс, активность) в веб-версии панели провайдера.
