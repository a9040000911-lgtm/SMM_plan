# Provider Link Requirements Skill (v2.0 - Data Driven)

> [!CAUTION]
> **КРИТИЧЕСКОЕ БИЗНЕС-ПРАВИЛО (WHITE-LABEL):**
> Для конечного клиента **Smmplan является единственным провайдером**. Категорически ЗАПРЕЩЕНО упоминать слова "провайдер", "поставщик", "партнеры" в любых клиентских интерфейсах (фронтенд, бот, описания услуг). Все требования сторонних API должны преподноситься как внутренние "Технические требования системы" или "Обязательные условия заказа".

Этот документ систематизирует реальные технические требования поставщиков SMM-услуг, найденные при анализе каталогов SocRocket, VexBoost и других.

## 1. Категория: Модификаторы и Строгие Форматы (URL Surgery)

Некоторые услуги (особенно Telegram Boosts) требуют использования строго специфических URL, отличных от стандартных ссылок на канал/пост.

### A. Telegram Boosts (Provider-Specific Formats)
Это самая "капризная" категория услуг. Неправильный формат = потеря денег.

- **Формат 1 (smmpanelus.com / Perfect Panel)**: 
  - `https://t.me/boost/username`
  - Требует путь `/boost/`. Строго.
- **Формат 2 (soc-rocket.ru / SocRocket)**: 
  - `https://t.me/username?boost`
  - Требует параметр запроса `?boost`. Строго.
- **Формат 3 (vexboost.ru / VexBoost)**: 
  - `https://t.me/username` ИЛИ `https://t.me/boost/username`
  - Гибкий формат. Принимает оба варианта.
- **Логика для SMM-панели**: 
  - При создании заказа сервис `link-analyzer` должен смотреть на `providerId`.
  - Для `soc-rocket` -> принудительно добавляем `?boost`. 
  - Для `smmpanelus` -> принудительно трансформируем в `/boost/`.
  - Для `vexboost` -> оставляем как есть (или приводим к стандарту `/boost/` для надежности).
  - **Автоматизация**: Клиент всегда вводит простую ссылку, а панель адаптирует её под "капризы" конкретного API.

### B. YouTube "Turbo/Boost"
- **Суть**: К стандартной ссылке видео добавляется `?boost`.
- **Зачем**: Информирует серверную ферму о необходимости использования специфических сессий.

### B. TikTok Cache Refresh
- **Суть**: Добавление `?v=1` или `?any_random_string`.
- **Зачем**: Обход кеширования TikTok CDN для обновления счетчиков в реальном времени.

## 2. Категория: Обязательные боты и Права администратора (Bot-as-Admin)

Для некоторых услуг (особенно в Discord и Telegram) недостаточно просто ссылки. Провайдеру требуется технический доступ к объекту.

### A. Discord (Online Members / Protection Bypass)
- **Услуга**: Онлайн-участники (SocRocket, SmmPanelUS).
- **Требование**: Пригласить сервисного бота (например, `offline.nowon.tools` или `onlybots.lol`) на сервер.
- **Критичность**: Без бота услуга либо не запустится, либо боты будут мгновенно кикнуты системой защиты Discord.
- **Действие в коде**: Если в `metadata` услуги есть флаг `requires_bot`, панель должна показывать модальное окно с инструкцией по приглашению бота.

### B. Telegram (Referrals & Bot Starts)
- **Услуга**: Рефералы в бот (VexBoost, SocRocket).
- **Формат ссылки**: `https://t.me/bot_username?start=ref_code`.
- **Требование**: Указывать именно реферальную ссылку, а не просто юзернейм бота.
- **Анализ**: `link-analyzer` должен проверять наличие параметра `?start=` для таких категорий.

### C. Telegram (Private Channels / Join Requests)
- **Услуга**: Подписчики на закрытые каналы с "заявками на вступление".
- **Требование**: Добавить бота провайдера в администраторы с правом "Add Members" или "Manage Messages".
- **Зачем**: Провайдер должен автоматически одобрять заявки ботов, которыми он накручивает канал.

## 3. Категория: Ограничения форматов (Format Enforcing)

Провайдеры часто запрещают сокращенные или пригласительные ссылки для определенных типов накрутки.

### A. Telegram "Search" Subscribers
- **Запрещено**: `t.me/+hash` (invite link).
- **Требуется**: Прямая ссылка `t.me/username`.
- **Причина**: Накрутка "из поиска" требует, чтобы бот вводил юзернейм в поисковую строку.

### B. YouTube Shorts vs Watch
- **Запрещено**: Ссылка `/shorts/ID` для обычных просмотров.
- **Трансформация**: Панель должна менять на `watch?v=ID`.

## 4. Категория: Валидация состояния (State Pre-check)

Требования к настройкам аккаунта клиента, без которых услуга уйдет в `Canceled`.

- **Open List**: Список подписчиков должен быть открыт (характерно для VK, OK).
- **No Verification**: На сервере Discord должны быть отключены все методы защиты и верификации (иначе боты не вступят).

## 5. Алгоритм для `link-analyzer.service.ts`

При реализации сервиса мы должны внедрить **Pipeline адаптации**:

1.  **Normalization**: Приводим ссылку к базовому виду (чистим мусор).
2.  **Provider Rules Check**: Загружаем `metadata` услуги из БД.
    - Если `metadata.requires_suffix === '?boost'`, добавляем суффикс (для SocRocket).
    - Если `metadata.requires_path === '/boost/'`, трансформируем путь (для SmmPanelUS).
    - Если `metadata.requires_bot === true`, проверяем наличие инструкции/ссылки на бота в описании и передаем её на фронтенд.
3.  **Validation**: 
    - Для Telegram Referrals проверяем наличие `?start=`.
    - Для Discord проверяем формат ссылки-приглашения.
4.  **Final Payload**: Формируем `link` параметр для API провайдера.
