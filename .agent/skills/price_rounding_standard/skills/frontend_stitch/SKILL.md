# Frontend Architecture (Stitch)

Данный навык описывает стандарты и процессы создания фронтенда для Smmplan. Методология "Stitch" (Сшивание) подразумевает сборку сложных интерфейсов из высококачественных, атомарных компонентов с упором на премиальную эстетику и производительность.

## Технологический стек (Stitch Core)
- **Framework**: Next.js 16 (App Router)
- **Engine**: React 19 (Server/Client separation)
- **Styling**: Tailwind CSS 4.0 (@theme based)
- **Animations**: Framer Motion
- **Icons**: Lucide React

## Структура страниц (Page Map)

Каждая страница должна "сшиваться" по следующему шаблону:
1. **Layout**: Обертка (Header + Footer + Sidebar).
2. **Hero/Header**: Акцентная область с заголовком и CTA.
3. **Body**: Контентные блоки (Widgets/Bento/Tables).
4. **Footer**: Информационная область.

### 1. Главная страница (`/`)
- **Назначение**: Первое касание, конверсия в быстрый заказ.
- **Компоненты**: Hero-секция с неоновым свечением, форма `QuickOrder`, блок "Как это работает", блок доверия (статистика в реальном времени).
- **Стиль**: Максимально "Wow"-эффект, использование стеклянного морфизма (Glassmorphism).

### 2. Каталог услуг (`/catalog`)
- **Назначение**: Просмотр всех доступных цен и услуг без авторизации.
- **Компоненты**: Платформенные табы (Instagram, TG, VK), аккордеоны категорий, фильтр поиска.
- **Особенность**: Динамическая подгрузка описаний через саннитайзер.

### 3. Дашборд пользователя (`/dashboard`)
- **Назначение**: Центр управления заказами и балансом.
- **Компоненты**: Статистическая Bento-сетка, список последних заказов, виджет текущих достижений.

### 4. Создание заказа (`/dashboard/orders/new`)
- **Назначение**: Реализация воронки заказа.
- **Компоненты**: Смарт-поиск, калькулятор цены, выбор параметров (Drip-feed).

### 5. Админ-панель (`/admin/*`)
- **Назначение**: Полный контроль системы.
- **Страницы**:
    - `admin/analytics` — Графики выручки и инцидентов.
    - `admin/services` — Редактирование маппингов и цен.
    - `admin/users` — Управление балансами и правами.
    - `admin/projects` — Настройка мульти-сайтов.

### 6. Авторизация (`/login`, `/register`)
- **Назначение**: Вход и регистрация.
- **Стиль**: Минимализм, фокус на форме, отсутствие лишних отвлекающих элементов.

### 7. Финансы и пополнение (`/dashboard/finance`)
- **Назначение**: Управление средствами.
- **Компоненты**: Выбор платежной системы (YooKassa, Crypto), история транзакций, калькулятор пополнения.

### 8. Реферальная программа (`/dashboard/referrals`)
- **Назначение**: Заработок на бонусах.
- **Компоненты**: Реферальная ссылка, статистика кликов/регистраций, таблица начислений.

### 9. Профиль и настройки (`/dashboard/profile`)
- **Назначение**: Кастомизация аккаунта.
- **Компоненты**: Смена пароля, привязка Telegram, настройки уведомлений.

### 10. Страницы ошибок и тех. работ (`404`, `500`, `maintenance`)
- **Назначение**: Обработка исключений и планового обслуживания.
- **Стиль**: Креативные иллюстрации, четкое описание проблемы и кнопка возврата на главную.

### 11. Результаты оплаты (`/payment/success`, `/payment/fail`)
- **Назначение**: Информирование о статусе транзакции после редиректа от эквайринга.
- **Компоненты**: Статус-иконка (Success/Error), детали заказа, кнопка перехода в дашборд.

### 12. Юридическая информация и FAQ (`/legal/[slug]`, `/faq`)
- **Назначение**: Ответы на вопросы и документы.
- **Реализация**: Контент хранится в `JSON` поле настроек проекта или как `Markdown` файлы. Фронтенд должен рендерить их через `react-markdown` с кастомными стилями.
- **Доступные слаги**: `rules`, `policy`, `refund`, `offer`.

## Взаимодействие с данными (Data Stitching)

### 1. Подключение к БД и Изоляция
Подключение осуществляется через Prisma Client. **Критически важно** всегда фильтровать данные по `projectId`, который передается через `process.env.PROJECT_ID`.

**Пример (Server Component):**
```tsx
import { prisma } from '@/lib/prisma';

export default async function Page() {
  const projectId = process.env.PROJECT_ID; // Привязка к конкретному фронтенду
  
  const services = await prisma.internalService.findMany({
    where: {
      isActive: true,
      lastProviderPrice: { gt: 0 },
      mappings: {
        some: { 
          projectId: projectId, // Гарантия изоляции!
          isActive: true 
        }
      }
    }
  });
  
  return <Catalog services={services} />;
}
```

### 2. Мутации (Server Actions)
- Все изменения (создание заказа, смена пароля) выполняются через **Server Actions**.
- Обязательная валидация через `zod` и проверка авторизации (`auth()`).

### 3. Обработка ошибок
- Использовать стандартизированные ошибки (напр. `SERVICE_UNAVAILABLE` при нулевой цене).
- Обратная связь пользователю через `sonner` или кастомные уведомлялки `Toast`.

## Локальная разработка (Multi-Frontend Support)

При параллельной разработке нескольких фронтендов (например, основной Smmplan и партнерский White Label), необходимо соблюдать следующие правила:

### 1. Управление портами
- Каждый проект должен иметь свой уникальный порт. По умолчанию:
  - Основной фронтенд: `3000`
  - Дополнительный: `3001`, `3002` и т.д.
- Запуск через `npm run dev -- -p <PORT>`.

### 2. Конфигурация окружения (`.env.local`)
- Для каждого локального фронтенда необходимо настраивать:
  - `NEXT_PUBLIC_URL`: Точный локальный адрес с портом (напр. `http://localhost:3001`).
  - `NEXT_PUBLIC_APP_URL`: Тот же адрес для консистентности ссылок.
  - `PROJECT_ID`: ID проекта из базы данных, чтобы фронтенд знал, какой контент "сшивать".

### 3. Тестирование мульти-проектности
- Для проверки изоляции данных рекомендуется открывать разные фронтенды в разных "инкогнито" окнах или использовать разные браузерные профили, чтобы избежать конфликтов сессий (cookies) на `localhost`.

## Визуальные стили (Design Archetypes)

Stitch поддерживает несколько предустановленных стилей для разных типов сайтов. Переключение стиля осуществляется через изменение цветовых токенов в `globals.css` или специфических Tailwind-классов.

### 1. Dark Premium (Neo-SMM)
- **Цель**: Максимальный визуальный эффект, молодежная аудитория.
- **Цвета**: `bg-slate-950`, `text-white`, `shadow-blue-500/20`.
- **Фишки**: Глубокие градиенты, неоновое свечение (Glow), стекломорфизм.
- **Пример**: [dark_premium.tsx](file:///d:/Smmplan/.agent/skills/frontend_stitch/examples/styles/dark_premium.tsx)

### 2. Light Clean (Breezy)
- **Цель**: Простота, доверие, легкость.
- **Цвета**: `bg-slate-50`, `text-slate-900`, `border-slate-200`.
- **Фишки**: Много "воздуха", мягкие тени (`shadow-sm`), пастельные акценты.
- **Пример**: [light_clean.tsx](file:///d:/Smmplan/.agent/skills/frontend_stitch/examples/styles/light_clean.tsx)

### 3. Corporate Modern (Professional)
- **Цель**: Для B2B, надежность, строгость.
- **Цвета**: `bg-white`, `text-indigo-950`, `accent-indigo-600`.
- **Фишки**: Строгая сетка, острые углы (`rounded-lg`), акцент на типографике.
- **Пример**: [corporate_modern.tsx](file:///d:/Smmplan/.agent/skills/frontend_stitch/examples/styles/corporate_modern.tsx)

## Принципы верстки (The Stitch Rules)

1. **Изоляция шрифтов**: Использовать `font-sans` и осторожно работать с `font-black`, чтобы избежать обрезки букв (использовать `overflow-visible` для заголовков).
2. **Responsive first**: Каждая страница должна идеально выглядеть на 375px и 1920px. Использовать `flex-col sm:flex-row`.
3. **No Placeholders**: Везде реальное данные или `generate_image`.
4. **Безупречность**: Не допускай ошибок. Код должен быть чистым, типизированным и проверенным.

## Автоматизация (Quick Start Scripts)

Для ускорения разработки рекомендуется использовать вспомогательные скрипты в директории `scripts/`:
- `stitch-page.sh`: Создает структуру `page.tsx` по шаблону Stitch.
- `stitch-component.sh`: Генерирует React компонент с предустановленными стилями Tailwind 4 и анимациями Framer Motion.
