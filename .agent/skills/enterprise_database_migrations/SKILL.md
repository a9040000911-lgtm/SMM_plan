# 🗄️ Enterprise Database Migrations (Zero-Downtime Standard)

Этот навык регулирует изменение структуры базы данных (Prisma Schema). В условиях высоконагруженных систем (HA) и Zero-Downtime деплоев, классический подход `prisma db push` или слепой `prisma migrate deploy` ведет к фатальным падениям серверов.

## 🔴 Фундаментальная Проблема "Гонки Контейнеров"
При Zero-Downtime деплое старый контейнер (V1) и новый (V2) работают параллельно в течение 2-5 секунд (пока Nginx переключает трафик).
Если релиз V2 удаляет колонку в БД, то через долю секунды старый контейнер V1 (все еще обслуживающий пользователей) попытается записать в неё данные. **Результат: HTTP 500 (Internal Server Error) для покупателя.**

## 🟢 Индустриальный Стандарт: The Expand-Contract Pattern
Любые деструктивные изменения схемы БД ЗАПРЕЩЕНЫ в рамках одного релиза. База данных должна быть всегда "Обратно-совместима" со старым кодом.
Процесс изменения колонки/таблицы делится на 3 фазы:

### Фаза 1: Expand (Расширение)
- В схему Prisma **добавляется** новая колонка (или таблица).
- Код приложения обновляется: он начинает читать/писать одновременно в старую И в новую колонку (Dual-Write).
- *Деплой Фазы 1.* (Безопасно, так как старая колонка на месте и V1 не ломается).

### Фаза 2: Migrate (Перенос Данных)
- Выполняется фоновый скрипт миграции старых данных из старой колонки в новую для исторических записей.
- Приложение продолжает использовать Dual-Write.

### Фаза 3: Contract (Сжатие)
- Код приложения перестает читать из старой колонки.
- Старая колонка **удаляется** из схемы Prisma.
- *Деплой Фазы 3.* (Безопасно, так как старый код V2, обслуживающий трафик, уже её не использует).

## 🛡️ Правила откатов (Rollback Mismatch Protocol)
- Никогда не откатывайте Docker-образ приложения (`docker update --image v1.0`), если в релизе `v1.1` была применена миграция БД.
- Откат кода без отката базы данных приведет к фатальной рассинхронизации Prisma.
- **Обязательное Правило:** Деплой-скрипт обязан создавать атомарный `pg_dump` базы ПЕРЕД запуском `prisma migrate`. В случае отката Docker-образа, база должна быть принудительно восстановлена в слепок до-деплоя (`pg_restore`).

> [!WARNING]
> Если в задаче требуется переименовать или удалить поле в Prisma, Агент обязан разбить задачу на два сабрелиза, используя паттерн Expand-Contract.
