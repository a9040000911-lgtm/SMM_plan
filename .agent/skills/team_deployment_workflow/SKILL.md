# Командный рабочий процесс и деплой

Этот навык регулирует совместную работу нескольких разработчиков (3+) над проектом Smmplan для обеспечения стабильности продакшена и целостности кода.

## 🏗️ 1. Стратегия окружения
- **Локально**: Каждый разработчик запускает `npm run dev`.
- **GitLab**: Центральный источник истины (`89.23.98.202:8080`).
- **Продакшен**: Живой сайт на том же сервере, управляемый через GitLab Runner.

## 👥 2. Безопасная разработка (Git Flow)
Чтобы не перезаписывать работу друг друга:
1. **Никогда не пушьте в `main` напрямую.**
2. **Работайте в ветках задач**: `git checkout -b feature/имя-задачи`.
3. **Ежедневная синхронизация**: Всегда делайте `git pull origin main` перед началом работы, чтобы избежать огромных конфликтов слияния.
4. **Merge Requests (MR)**: Создавайте MR в GitLab для каждой задачи.
   - Требуется как минимум 1 подтверждение ("Approve") от коллеги.
   - Пайплайн CI/CD должен пройти успешно (зеленая галочка).

## 🚀 3. Протокол деплоя (Hybrid Build)
Мы используем **гибридную стратегию** (Локальная сборка -> Удаленный образ) для экономии оперативной памяти сервера.

### Шаги для деплоя:
1. **Проверка**: Запустите `npm run build` локально. Если сборка упала — исправьте код.
2. **Коммит и Пуш**: `git commit -m "..." && git push origin main`.
3. **Автоматизация**: GitLab Runner обнаружит пуш и запустит `scripts/migrate-server.sh`.

### Ручной деплой (Резервный):
Если CI/CD не сработал, используйте подготовленный мной скрипт ручного переноса:
```powershell
./scripts/deploy-production.ps1
```

## 🛠️ 4. Управление GitLab Runner
Раннер выступает мостом между GitLab и Docker на сервере.

- **Проверить статус**: `docker ps | grep runner`
- **Регистрация нового раннера**:
  ```bash
  docker exec -it gitlab-runner gitlab-runner register \
    --non-interactive \
    --url "http://89.23.98.202:8080/" \
    --registration-token "ВАШ_ТОКЕН_ИЗ_GITLAB" \
    --executor "docker" \
    --docker-image docker:latest \
    --description "smmplan-shell-runner" \
    --docker-volumes "/var/run/docker.sock:/var/run/docker.sock"
  ```

## ⚠️ 5. Чек-лист безопасности
- [ ] **Typecheck**: Отсутствие ошибок TypeScript (`npx tsc --noEmit`).
- [ ] **Lint**: Отсутствие ошибок ESLint (`npm run lint`).
- [ ] **PRISMA**: Всегда обновляйте схему: `npx prisma generate`, а затем `npx prisma db push` во время деплоя.
- [ ] **.env**: НИКОГДА не коммитьте ключи доступа. Используйте GitLab CI Variables для секретов продакшена.

> [!IMPORTANT]
> Задача считается выполненной ТОЛЬКО тогда, когда она работает и проверена на сервере.
