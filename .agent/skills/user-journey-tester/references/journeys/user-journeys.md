## Journey 1: Регистрация и первый вход

### Happy Path
```
1. Пользователь открывает страницу регистрации
   └─→ Форма загружается корректно ✅

2. Вводит email: user@example.com
   └─→ Валидация email проходит ✅

3. Вводит пароль: SecurePass123!
   └─→ Пароль соответствует требованиям ✅

4. Подтверждает пароль
   └─→ Пароли совпадают ✅

5. Нажимает "Зарегистрироваться"
   └─→ Создаётся аккаунт ✅
   └─→ Отправляется verification email ✅

6. Переходит по ссылке из email
   └─→ Email подтверждается ✅
   └─→ Автоматический вход ✅

7. Попадает в дашборд
   └─→ Видит приветственное сообщение ✅
   └─→ Баланс = 0 ✅
```

### Error Paths
| Сценарий | Ввод | Ожидается | Проверка |
|----------|------|-----------|----------|
| Email уже существует | existing@email.com | "Email уже зарегистрирован" | ✅/❌ |
| Неверный формат email | notanemail | "Неверный формат email" | ✅/❌ |
| Слабый пароль | 123 | "Минимум 8 символов" | ✅/❌ |
| Пароли не совпадают | Pass1 ≠ Pass2 | "Пароли не совпадают" | ✅/❌ |
| Email не пришёл | — | Кнопка "Отправить повторно" | ✅/❌ |
| Ссылка истекла | Старый токен | "Ссылка истекла, запросите новую" | ✅/❌ |

### Edge Cases
| Случай | Тест | Ожидается |
|--------|------|-----------|
| Unicode в email | пользователь@почта.рф | Корректная обработка |
| Пробелы в email | " user@email.com " | Trim и принятие |
| Очень длинный пароль | 1000 символов | Ограничение длины |
| Emoji в пароле | "Pass🔒123" | Принятие или валидное сообщение |

## Journey 2: Создание Single-заказа

### Happy Path
```
1. Пользователь авторизован (баланс > 0)
   └─→ Сессия активна ✅

2. Переходит в "Каталог услуг"
   └─→ Список услуг загружается ✅

3. Выбирает категорию "Instagram"
   └─→ Фильтрация работает ✅

4. Выбирает услугу "Подписчики [HQ]"
   └─→ Открывается форма заказа ✅

5. Вводит ссылку: https://instagram.com/username
   └─→ Ссылка валидна ✅
   └─→ Определяется тип: профиль ✅

6. Вводит количество: 1000
   └─→ Количество в пределах min/max ✅

7. Видит рассчитанную стоимость: 50 RUB
   └─→ Расчёт корректен ✅

8. Нажимает "Создать заказ"
   └─→ Баланс списывается ✅
   └─→ Заказ создаётся (статус: PENDING) ✅
   └─→ Отправляется провайдеру ✅

9. Видит подтверждение
   └─→ "Заказ #12345 создан" ✅
   └─→ Редирект на страницу заказа ✅

10. Отслеживает статус
    └─→ Статус обновляется в реальном времени ✅
```

### Error Paths
| Сценарий | Ожидается | Критичность |
|----------|-----------|-------------|
| Недостаточно средств | "Пополните баланс на X RUB" + кнопка | Высокая |
| Невалидная ссылка | "Проверьте ссылку на профиль" | Средняя |
| Приватный профиль | "Профиль должен быть открыт" | Средняя |
| Несуществующий профиль | "Профиль не найден" | Средняя |
| Услуга недоступна | "Временно недоступно" + уведомление | Низкая |
| Количество < min | "Минимум: 50 подписчиков" | Низкая |
| Количество > max | "Максимум: 10000 подписчиков" | Низкая |
| Провайдер не отвечает | Заказ в PENDING, retry через 30 сек | Высокая |

### Edge Cases
| Случай | Тест | Ожидается |
|--------|------|-----------|
| Ссылка с www | www.instagram.com/user | Принимается |
| Мобильная ссылка | m.instagram.com/user | Принимается |
| Ссылка с параметрами | instagram.com/user?param=1 | Очищается |
| Количество = min | 50 | Принимается |
| Количество = max | 10000 | Принимается |
| Граничное (min - 1) | 49 | Ошибка валидации |
| Граничное (max + 1) | 10001 | Ошибка валидации |

## Journey 3: Пополнение баланса

### Happy Path (через платёжную систему)
```
1. Пользователь переходит в "Баланс"
   └─→ Видит текущий баланс ✅

2. Нажимает "Пополнить"
   └─→ Открывается форма ✅

3. Выбирает валюту: RUB
   └─→ Доступные методы обновляются ✅

4. Выбирает способ: "Банковская карта"
   └─→ Открывается шлюз ✅

5. Вводит сумму: 1000 RUB
   └─→ Валидация суммы ✅

6. Нажимает "Оплатить"
   └─→ Редирект на платёжный шлюз ✅

7. Оплачивает картой
   └─→ Платёж проходит ✅

8. Возвращается в панель
   └─→ Баланс обновился ✅
   └─→ Транзакция в истории ✅
   └─→ Уведомление об успехе ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| Сумма < минимальной | "Минимальная сумма: 100 RUB" |
| Сумма > максимальной | "Максимальная сумма: 50000 RUB" |
| Платёж отклонён | "Ошибка оплаты, попробуйте другой способ" |
| Таймаут платёжного шлюза | "Проверка статуса..." + retry |
| Повторный платёж (double click) | Защита от дублей |

## Journey 4: Drip-Feed заказ

### Happy Path
```
1. Пользователь выбирает услугу
   └─→ Включает toggle "Drip-Feed" ✅

2. Настраивает параметры:
   - Общее количество: 1000
   - Заказов: 10
   - Интервал: 60 минут
   └─→ Расчёт корректен ✅

3. Создаёт заказ
   └─→ Создаётся родительский заказ ✅
   └─→ Создаются 10 дочерних заказов ✅
   └─→ Дочерние в статусе SCHEDULED ✅

4. Система выполняет по расписанию
   └─→ Каждый час активируется заказ ✅
   └─→ Статусы обновляются ✅

5. Пользователь видит прогресс
   └─→ Выполнено 5/10 ✅
   └─→ Осталось 500/1000 ✅
```

### Error Paths
| Сценарий | Проблема | Решение |
|----------|---------|---------|
| Недостаточно средств на все заказы | Проверка полной суммы upfront | Предупреждение |
| Баланс уменьшился во время drip | Ошибка при активации заказа | Пауза + уведомление |
| Услуга стала недоступна | Fail на запланированном заказе | Пропуск + уведомление |

## Journey 5: Реферальная программа

### Happy Path
```
1. Пользователь переходит в "Рефералы"
   └─→ Видит свою реферальную ссылку ✅

2. Копирует ссылку
   └─→ Скопировано в буфер ✅

3. Приглашает друга
   └─→ Друг регистрируется по ссылке ✅
   └─→ friend.referrer_id = user.id ✅

4. Друг делает первый заказ
   └─→ Пользователь получает бонус ✅
   └─→ Бонус на реферальный баланс ✅

5. Пользователь использует реферальный баланс
   └─→ Часть оплаты из рефералов ✅
   └─→ Остаток с основного баланса ✅
```

### Multi-level проверка
```
Уровень 1: Прямой реферал → 10% от заказов
Уровень 2: Реферал реферала → 5% от заказов
Уровень 3: Реферал 3-го уровня → 2% от заказов

Проверить:
- [ ] Корректный расчёт процентов
- [ ] Зачисление на нужный баланс
- [ ] История реферальных начислений
- [ ] Лимиты на вывод реферального баланса
```

## Journey 6: Telegram Bot

### Happy Path
```
1. Пользователь находит бота в Telegram
   └─→ Бот доступен ✅

2. Отправляет /start
   └─→ Бот приветствует ✅
   └─→ Предлагает привязать аккаунт ✅

3. Нажимает "Привязать аккаунт"
   └─→ Бот просит email ✅

4. Вводит email от панели
   └─→ Бот отправляет код на email ✅

5. Вводит код из email
   └─→ Аккаунт привязан ✅
   └─→ Telegram ID сохранён ✅

6. Создаёт заказ через бота
   └─→ Wizard-сценарий работает ✅
   └─→ Заказ создаётся ✅
   └─→ Уведомление о статусе ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| Email не найден | "Аккаунт не найден" |
| Неверный код | "Неверный код, попробуйте снова" |
| Код истёк | "Код истёк, запросите новый" |
| Аккаунт уже привязан | "Этот Telegram уже привязан" |
