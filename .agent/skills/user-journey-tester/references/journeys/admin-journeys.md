# User Journeys: ADMIN & SUPERADMIN

Полный набор сценариев для административных ролей в SMM-панели.

---

## 📋 Различия ролей

| Возможность | ADMIN | SUPERADMIN |
|-------------|-------|------------|
| Просмотр заказов своего проекта | ✅ | ✅ |
| Просмотр ВСЕХ заказов | ❌ | ✅ |
| Управление услугами проекта | ✅ | ✅ |
| Создание новых проектов | ❌ | ✅ |
| Назначение админов проектов | ❌ | ✅ |
| Глобальные настройки системы | ❌ | ✅ |
| Просмотр системных логов | ❌ | ✅ |
| Управление провайдерами | ❌ | ✅ |
| Блокировка пользователей | Проекта | Глобально |

## Journey A1: Вход в админ-панель

### Happy Path (ADMIN)
```
1. Администратор входит под своими учётными данными
   └─→ Система проверяет роль ✅

2. Система определяет проекты администратора
   └─→ projectIds = [1, 3, 5] ✅

3. Открывается дашборд
   └─→ Видна статистика только своих проектов ✅
   └─→ Нет доступа к другим проектам ✅

4. Администратор выбирает проект
   └─→ Переключение контекста ✅
   └─→ Данные фильтруются по проекту ✅
```

### Happy Path (SUPERADMIN)
```
1. Суперадмин входит в систему
   └─→ Роль = SUPERADMIN ✅

2. Открывается глобальный дашборд
   └─→ Видна статистика ВСЕХ проектов ✅
   └─→ Доступ ко всем разделам ✅

3. Может переключаться между проектами
   └─→ Выпадающий список всех проектов ✅
   └─→ Или режим "Все проекты" ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| ADMIN пытается доступ к чужому проекту | 403 Forbidden |
| ADMIN заблокирован | "Доступ запрещён" + редирект |
| У ADMIN нет проектов | "Вам не назначен ни один проект" |
| Сессия истекла | Редирект на логин |

## Journey A2: Управление заказами

### Happy Path (просмотр)
```
1. Администратор открывает "Заказы"
   └─→ Список заказов проекта ✅

2. Применяет фильтры:
   - По статусу: IN_PROGRESS, PENDING
   - По дате: последние 7 дней
   - По услуге: конкретная услуга
   └─→ Фильтрация работает ✅

3. Открывает карточку заказа
   └─→ Все данные заказа ✅
   └─→ История статусов ✅
   └─→ Информация о провайдере ✅
   └─→ Связанные сущности ✅
```

### Happy Path (изменение статуса)
```
1. Администратор выбирает заказ
   └─→ Открывает действия ✅

2. Выбирает "Изменить статус"
   └─→ Доступные статусы ✅

3. Выбирает "REFUND"
   └─→ Запрашивается причина ✅

4. Вводит причину: "По просьбе клиента"
   └─→ Подтверждение действия ✅

5. Подтверждает
   └─→ Статус меняется ✅
   └─→ Средства возвращаются на баланс ✅
   └─→ Транзакция создаётся ✅
   └─→ Уведомление пользователю ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| Недопустимый переход статуса | "Нельзя изменить с COMPLETE на PENDING" |
| Refund при нулевом балансе провайдера | Предупреждение о невозможности |
| Попытка изменить чужой заказ | 403 Forbidden (для ADMIN) |
| Параллельное изменение (race condition) | Optimistic lock / перезагрузка |

## Journey A3: Управление услугами

### Happy Path (создание услуги)
```
1. Администратор открывает "Услуги"
   └─→ Список услуг проекта ✅

2. Нажимает "Добавить услугу"
   └─→ Форма создания ✅

3. Заполняет данные:
   - Название: "Instagram Подписчики HQ"
   - Категория: "Instagram"
   - Мин. количество: 50
   - Макс. количество: 10000
   - Цена за 1000: 50 RUB
   └─→ Валидация проходит ✅

4. Настраивает маппинг провайдера:
   - Провайдер: "SMMProvider1"
   - ID услуги провайдера: 123
   └─→ Связь создаётся ✅

5. Сохраняет
   └─→ Услуга создаётся ✅
   └─→ Появляется в каталоге ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| Дублирующееся название | "Услуга с таким названием уже существует" |
| Неверный ID провайдера | Проверка через API провайдера |
| Цена <= 0 | "Цена должна быть больше 0" |
| min > max | "Минимум не может быть больше максимума" |

## Journey A4: Управление провайдерами (SUPERADMIN)

### Happy Path (добавление провайдера)
```
1. Суперадмин открывает "Провайдеры"
   └─→ Список всех провайдеров ✅

2. Нажимает "Добавить провайдера"
   └─→ Форма создания ✅

3. Заполняет данные:
   - Название: "BestSMM"
   - API URL: "https://api.bestsmm.com/v1"
   - API Key: "xxxxx"
   - Тип: " стандартный API"
   └─→ Валидация ✅

4. Нажимает "Проверить соединение"
   └─→ Тестовый запрос к API ✅
   └─→ "Соединение успешно" ✅

5. Сохраняет
   └─→ Провайдер создаётся ✅
   └─→ Можно использовать в услугах ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| Неверный API URL | "Не удалось подключиться" |
| Неверный API Key | "Ошибка авторизации" |
| Дубликат названия | "Провайдер с таким названием существует" |
| Timeout при проверке | "Превышено время ожидания" |

## Journey A5: Финансовые отчёты

### Happy Path
```
1. Администратор открывает "Финансы"
   └─→ Отчёт по проекту(ам) ✅

2. Выбирает период: "Этот месяц"
   └─→ Данные обновляются ✅

3. Видит ключевые метрики:
   - Пополнения: 500,000 RUB
   - Расходы на провайдеров: 300,000 RUB
   - Маржа: 200,000 RUB (40%)
   - Количество заказов: 1,234
   └─→ Расчёты корректны ✅

4. Скачивает отчёт в Excel
   └─→ Файл формируется ✅
   └─→ Все данные включены ✅
```

### Cross-check (проверка целостности)
```
Требуется проверить:
- [ ] Сумма пополнений = сумма транзакций с type=DEPOSIT
- [ ] Сумма списаний = сумма заказов по закупочной цене
- [ ] Маржа = Пополнения - Расходы
- [ ] Балансы пользователей = сумма всех balance
- [ ] Нет orphaned транзакций без заказа
- [ ] Нет заказов без транзакции
```

## Journey A6: Управление пользователями

### Happy Path (просмотр)
```
1. Администратор открывает "Пользователи"
   └─→ Список пользователей проекта ✅

2. Применяет фильтры:

   - Баланс: > 1000 RUB
   - Статус: активные
   └─→ Фильтрация работает ✅

3. Открывает профиль пользователя
   └─→ Контактные данные ✅
   └─→ История заказов ✅
   └─→ Баланс и транзакции ✅
   └─→ Рефералы ✅
```

### Happy Path (блокировка)
```
1. Администратор находит проблемного пользователя
   └─→ Открывает профиль ✅

2. Нажимает "Заблокировать"
   └─→ Запрашивается причина ✅

3. Вводит причину: "Мошенничество"
   └─→ Подтверждение ✅

4. Подтверждает
   └─→ Пользователь блокируется ✅
   └─→ Сессии инвалидятся ✅
   └─→ Новые заказы запрещены ✅
   └─→ Лог создаётся ✅
```

   └─→ Переход к проблеме ✅
   └─→ Возможные действия ✅

## Journey A8: Массовые операции

### Happy Path (массовое обновление статусов)
```
1. Суперадмин открывает список заказов
   └─→ Выбирает фильтр: FAILED ✅

2. Выбирает заказы чекбоксами
   └─→ Выбрано 25 заказов ✅

3. Нажимает "Массовые действия"
   └─→ Список действий ✅

4. Выбирает "Перезапустить"
   └─→ Подтверждение ✅

5. Подтверждает
   └─→ Заказы переходят в PENDING ✅
   └─→ Retry механизм запускается ✅
   └─→ Уведомления отправлены ✅
```

### Error Paths
| Сценарий | Ожидается |
|----------|-----------|
| Выбраны заказы разных проектов (ADMIN) | "Можно выбрать только заказы одного проекта" |
| Слишком много заказов (> 100) | "Максимум 100 заказов за раз" |
| Недостаточно прав | Действие не доступно |
