# Cross-Role Scenarios: Кросс-ролевое тестирование

Сценарии взаимодействия между различными ролями в SMM-панели.

---

## 🔄 Общая матрица взаимодействий

```
               USER    ADMIN   SUPER   SUPPORT
USER           —       ✅      ✅      ✅
ADMIN          ✅      —       ✅      ✅
SUPERADMIN     ✅      ✅      —       ✅
SUPPORT        ✅      ✅      ✅      —

Примеры:
USER → ADMIN: жалоба на услугу
ADMIN → USER: возврат средств
SUPERADMIN → ADMIN: назначение на проект
```

---

## Scenario C1: User создает заказ → Admin видит и обрабатывает

### Happy Path
```
═══════════════════════════════════════════════════════════
 USER SIDE                          ADMIN SIDE
═══════════════════════════════════════════════════════════

1. USER создаёт заказ #12345
   └─→ Статус: PENDING
   └─→ Списание баланса: -50 RUB
   └─→ Уведомление: "Заказ создан"
   
2.                         ADMIN открывает список заказов
                          └─→ Видит заказ #12345 ✅
                          └─→ Данные корректны ✅

3.                         ADMIN видит проблему:
                          "Провайдер не отвечает"

4.                         ADMIN меняет провайдера вручную
                          └─→ Переназначение ✅

5.                         Заказ отправляется новому провайдеру
                          └─→ Статус: IN_PROGRESS ✅

6. USER видит изменение статуса
   └─→ "В обработке" ✅
   
7. Заказ выполняется
   └─→ Статус: COMPLETED ✅
   └─→ Уведомление USER ✅

═══════════════════════════════════════════════════════════
```

### Что проверить
- [ ] Время появления заказа в админ-панели (< 1 сек)
- [ ] Корректность данных заказа
- [ ] Уведомления пользователю о смене статуса
- [ ] Логирование действий админа
- [ ] Атомарность операций

## Scenario C2: User жалуется → Support обрабатывает → Admin решает

### Happy Path
```
═══════════════════════════════════════════════════════════
 USER          SUPPORT           ADMIN           SYSTEM
═══════════════════════════════════════════════════════════

1. USER создаёт тикет #100
   Тема: "Заказ не выполняется"
   Прикрепляет: скриншот
   
2.        SUPPORT получает уведомление
         └─→ Видит новый тикет ✅

3.        SUPPORT отвечает:
         "Проверяю ваш заказ..."

4.        SUPPORT смотрит заказ #12345
         └─→ Видит проблему:
         "Провайдер завис"

5.        SUPPORT эскалирует админу
         └─→ Назначает ADMIN ✅

6.                          ADMIN получает уведомление
                           └─→ Видит эскалацию ✅

7.                          ADMIN принимает решение:
                           Refund + бонус 10%

8.                          ADMIN оформляет refund
                           └─→ Возврат средств ✅

9.        SUPPORT видит решение
         └─→ Сообщает USER ✅

10. USER получает уведомление
    └─→ Баланс: +55 RUB ✅
    └─→ Тикет закрыт ✅

═══════════════════════════════════════════════════════════
```

### Error Paths
| Сценарий | Кто | Ожидается |
|----------|-----|-----------|
| Тикет без ответа > 24ч | SYSTEM | Автоэскалация |
| ADMIN не реагирует | SUPERADMIN | Уведомление |
| USER не доволен решением | SUPPORT | Повторное открытие |
| Прикреплён большой файл | USER | "Максимум 10MB" |

## Scenario C3: User пополняет баланс → Admin видит транзакцию

### Happy Path
```
═══════════════════════════════════════════════════════════
 USER              PAYMENT          ADMIN           DB
═══════════════════════════════════════════════════════════

1. USER инициирует платёж
   Сумма: 1000 RUB
   Метод: Card
   
2.        PAYMENT редиректит
         на шлюз

3.        PAYMENT обрабатывает
         └─→ Success callback ✅

4.        PAYMENT webhook
         на сервер

5.                                          DB обновляет
                                          └─→ balance += 1000
                                          └─→ transaction created

6. USER видит обновление
   └─→ Баланс: 1000 RUB ✅

7.                          ADMIN открывает "Финансы"
                           └─→ Видит транзакцию ✅
                           └─→ Данные верны ✅

8.                          ADMIN сверяет с платёжным
                           шлюзом
                           └─→ IDs совпадают ✅

═══════════════════════════════════════════════════════════
```

### Race Conditions для проверки
```
Сценарий: Двойной webhook от платёжной системы

1. Webhook 1 приходит → обработка
2. Webhook 2 приходит (параллельно) → ???

Ожидается:
- Idempotency key проверка
- Только одно зачисление
- Логирование дубля

Проверить:
- [ ] При параллельных вебхуках баланс увеличивается только 1 раз
- [ ] Второй webhook логируется как duplicate
- [ ] Нет orphaned транзакций
```

## Scenario C5: User достигает уровня лояльности → Система начисляет бонус

### Happy Path
```
═══════════════════════════════════════════════════════════
 USER              LOYALTY          DB               ADMIN
═══════════════════════════════════════════════════════════

1. USER делает заказ на 5000 RUB
   └─→ Общая сумма заказов: 10,000 RUB

2.        LOYALTY проверяет достижения
         └─→ "Оптовый покупатель"
```

## Scenario C6: Мультипроектная изоляция данных

### Проверка изоляции
```
═══════════════════════════════════════════════════════════
 ADMIN_A (Проект 1)          ADMIN_B (Проект 2)
═══════════════════════════════════════════════════════════

1. ADMIN_A создаёт заказ #1 в проекте 1
   └─→ Заказ виден только в проекте 1 ✅

2.                          ADMIN_B открывает заказы
                           └─→ Не видит заказ #1 ✅
                           └─→ Видит только проект 2 ✅

3. ADMIN_A пытается API-запрос:
   GET /api/orders?projectId=2
   
   Ожидается: 403 Forbidden ✅

4.                          ADMIN_B пытается:
                           GET /api/users?projectId=1
                           
                           Ожидается: 403 Forbidden ✅

═══════════════════════════════════════════════════════════
```

### Тесты безопасности
| Тест | Метод | Ожидается |
|------|-------|-----------|
| Прямой доступ к чужому заказу | GET /api/orders/123 | 403 если другой проект |
| Изменение чужого пользователя | PATCH /api/users/456 | 403 если другой проект |
| SQL injection через projectId | GET /api/orders?projectId=1 OR 1=1 | Валидация + ошибка |
| IDOR через заказ | GET /api/orders/123 | Проверка project ownership |

## Чек-лист кросс-ролевого тестирования

### User → Admin
- [ ] Заказ появляется в админ-панели мгновенно
- [ ] Пополнение отражается в финансовых отчётах
- [ ] Жалобы доходят до поддержки
- [ ] Реферальные начисления видны админу

### Admin → User
- [ ] Refund корректно зачисляется
- [ ] Изменение статуса видно пользователю
- [ ] Бонусы начисляются
- [ ] Уведомления приходят

### SuperAdmin → Admin
- [ ] Назначение на проект работает
- [ ] Снятие с проекта работает
- [ ] Ограничения прав соблюдаются

### Support → All
- [ ] Тикеты видны всем участникам
- [ ] История переписки сохраняется
- [ ] Эскалации доходят

### Изоляция данных
- [ ] ADMIN видит только свой проект
- [ ] USER видит только свои данные
- [ ] Нет утечек между проектами
- [ ] API endpoints защищены

### Race Conditions
- [ ] Параллельные платежи
- [ ] Параллельные изменения заказа
- [ ] Параллельные рефанды
- [ ] Оптимистичные блокировки
