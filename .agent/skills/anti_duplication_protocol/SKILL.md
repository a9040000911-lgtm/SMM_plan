# 🚨 АНТИ-ДУБЛИРОВАНИЕ И ЕДИНАЯ АРХИТЕКТУРА (ANTI-DUPLICATION PROTOCOL) 🚨

Этот протокол **СТРОГО ОБЯЗАТЕЛЕН** для ИИ-агента перед написанием любого кода, созданием новых файлов или папок в проекте `Smmplan`.

## Основная проблема
Из-за ограниченного контекста ИИ часто создает дубликаты существующих компонентов (например, вторую кнопку `Button.tsx`, хотя она уже есть в `src/components/ui/`) или плодит лишние папки. Это нарушает принцип DRY (Don't Repeat Yourself) и засоряет кодовую базу.

## 🛠️ Обязательный алгоритм действий (The Verification Loop)

ПЕРЕД тем, как агент создаст новый файл или папку, он **ОБЯЗАН** выполнить следующие шаги:

### 1. Разведка структуры (Directory Census)
- **ЗАПРЕЩЕНО** создавать новые папки без предварительного изучения родительской директории через инструмент `list_dir`.
- Если агент хочет создать фичу для панели администратора, он должен проверить существующую структуру: `list_dir("src/components/admin")` или `list_dir("src/app/admin")`.

### 2. Семантический поиск существующих решений (Semantic Search)
- **ЗАПРЕЩЕНО** писать новые UI-компоненты, хуки или утилиты без предварительного поиска по кодовой базе.
- Агент **ОБЯЗАН** использовать инструмент `grep_search`. 
- *Пример:* Если нужна кнопка загрузки, агент сначала ищет `grep_search(SearchPath: "src/components", Query: "Button")` или `grep_search(SearchPath: "src/components", Query: "Loading")`.

### 3. Приоритет переиспользования и расширения (Reusability Mandate)
- Если найден похожий компонент, кнопка, инпут или сервис — агент **ОБЯЗАН** переиспользовать его.
- Если существующему функционалу не хватает гибкости, агент должен **расширить** его интерфейс (добавить новые props, например `variant="outline"`), сохранив обратную совместимость, а не создавать копию.

### 4. Отражение в плане (Planning Mode Enforcement)
В файле `implementation_plan.md` агент обязан добавить раздел:
**"Проверка на дублирование:"**
И явно написать: 
* "Я проверил через `list_dir` и `grep_search` — компонента X нет, поэтому я создаю новый." ИЛИ
* "Я нашел компонент Y, буду использовать/расширять его."

## Структурные конвенции Smmplan (Куда что класть)
* **`src/components/ui/`** — только переиспользуемые, "глупые" базовые компоненты (кнопки, инпуты, модальные окна, карточки). Никакой бизнес-логики.
* **`src/lib/utils/`** — общие хелперы (форматирование дат, цен, строк). Не создавайте новые файлы, если можно добавить функцию в существующие (например, в `utils.ts`).
* **`src/lib/services/`** — классы логики, работающие с БД (Server Actions / Prisma). 
* **`src/components/[домен]/`** — компоненты, специфичные для конкретного раздела (например `src/components/admin/support/`).

Если агент нарушает этот протокол — пользователь имеет право отклонить план (Reject) с комментарием "Нарушен Anti-Duplication Protocol".
