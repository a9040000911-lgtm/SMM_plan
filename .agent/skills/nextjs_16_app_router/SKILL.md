# Next.js 16 Framework Rules

Smmplan строится на **Next.js 16.0.10** (App Router).

## 1. App Router Architecture
- Любой роутинг происходит через директорию `src/app`.
- Разделяйте Route Groups (в скобках) для группировки Layouts: `(client)` для пользовательской части, `(admin)` для админки. Маршрутизация при этом не меняется (например, `src/app/(client)/catalog` доступен по URL `/catalog`).
- Изолируйте сложную логику страниц, создавая `<Feature>UI.tsx` компоненты и импортируя их в `page.tsx`. `page.tsx` должен быть минимальным серверным компонентом (только fetch данных и передача props).

## 2. Server Actions
- Бизнес-логика обработки форм и мутаций БД должна лежать в изолированных файлах внутри `src/app/actions` или `src/services`, помеченных `"use server"`.
- Не пишите логику работы с базой данных внутри клиентских компонентов. Клиент только вызывает Action.

## 3. Caching & Revalidation (Кэширование)
- В Next.js 16 кэширование агрессивнее. При изменении данных в базе вы **ОБЯЗАНЫ** очистить кэш, иначе пользователи будут видеть старые данные.
- Используйте `revalidatePath('/path')` (для очистки кэша страницы) или `revalidateTag('my_tag')` (для очистки конкретного fetch-запроса). Вызывайте их прямо в Server Actions после `await prisma...`.

## 4. Metadata API (SEO)
- Обязательно экспортируйте объект `metadata` или функцию `generateMetadata` из `page.tsx` и `layout.tsx` для SEO-оптимизации.

## 5. Turbopack
- Использование Turbopack означает, что конфигурационные файлы (next.config) и CSS обрабатываются иначе. Избегайте нестандартных Webpack плагинов. Используйте нативные решения.

**Архитектура Smmplan настаивает на максимальном использовании Server Components для минимизации JS-бандла для конечного клиента.**
