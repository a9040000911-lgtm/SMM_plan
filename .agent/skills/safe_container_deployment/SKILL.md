# 🛡️ Enterprise Container Deployment (Deep Ops Protocol)

Этот навык не просто описывает деплой, но и решает фундаментальные технические изъяны Node.js и Next.js при горячей замене. Базируется на стандартах **Google SRE** и **GitLab Edge DevOps**.

## 1. Запрет Серверных Сборок (The OOM-Killer Prevention)
**Тезис:** Сервер `89.23.98.202` (Production) — это Execution-среда. Не Compilation-среда.
Команда `docker-compose build` на сервере потребляет гигабайты ОЗУ. При скачке пользовательского трафика база данных и ядро ОС будут заморожены.
**Стандарт:** Сборка осуществляется изолированно (например, через Github Actions). Собирается бинарный Docker-образ (`smmplan:vX.Y.Z`) и размещается в **GHCR** (Github Container Registry). Роль сервера — скачивать легкие готовые образы (20 мегабайт дельты) и применять их.

## 2. Генетика Горячей Замены (Zero-Downtime Edge Cases)
Если просто перезапустить контейнер, приложение "мигнёт". Для клиента это означает потерянный запрос или сбой транзакции. 
Разработчик ОБЯЗАН интегрировать архитектуру под следующие законы физики сетей:

### А. Синдром "Обрубленного Соединения" (SIGTERM Graceful Exit)
По умолчанию Node.js игнорирует сигнал `SIGTERM` от Docker'a при деплое. Приложение убивается топором через 10 секунд по `SIGKILL`.
**Решение:** В кастомном `server.js` (или Next.js Custom Server) должен быть слушатель:
```javascript
process.on('SIGTERM', () => {
  server.close(() => process.exit(0)); // Закрываем приём новых хитов, дорабатываем старые и выключаемся.
});
```

### Б. Синдром "Взрыва Процессора" (Next.js Cache Eviction)
При рестарте контейнера внутренний кэш Next.js (папка `.next/cache`) стирается. Когда на чистый контейнер обрушивается 1000 клиентов, Next.js начинает генерировать все страницы с нуля, загружая CPU на 100%. Остановка сайта.
**Решение:** Кэш должен быть **бессмертным**. Используется либо подключение Docker Volume (`/var/next_cache:/app/.next/cache`), либо вынос кэш-провайдера во внешний **Redis** (Custom Next.js Cache Handler).

## 3. Фильтр Грязной Крови (Husky & Secret Scanning)
Людской фактор — главная уязвимость систем. Защита встраивается на компьютер разработчика.
- **Инструментарий:** `husky` + `lint-staged`. Хуки `pre-commit` и `pre-push`.
- **Gitleaks:** Каждое сохранение сканируется антивирусом ключей. Токены API Telegram, YooKassa категорически запрещено хардкодить. Система должна физически заблокировать `git push` при обнаружении паттерна токена.
- **Log Masking:** В CI/CD пайплайнах (Actions) используется синтаксис `echo "::add-mask::$SECRET"` чтобы скрыть переменные от распечатки в логах системы.

## 4. Разбор Инцидентов: Корневая Причина 502 Bad Gateway
Во время Soft-Switching выкатки (Zero-Downtime) часто возникает фатальная ошибка **502 Bad Gateway**, когда Nginx не может достучаться до Node.js. Глубокий анализ показал, что причина НЕ в Nginx, а в архитектурном "Рассинхроне":

**Сценарий Аварии:**
1. Выкатывается новый Docker-контейнер `smmplan_app`.
2. Код приложения содержит новый Prisma-запрос (например, выборка новой колонки `organizationId`).
3. При старте контейнера, Node.js проверяет соединение с БД. При обращении к отсутствующему столбцу Prisma ловит `P2022: Column does not exist`.
4. Контейнер `smmplan_app` крашится и уходит в статус `Restarting`.
5. Nginx перенаправляет клиентский трафик на порт 3000, но контейнер мёртв. Nginx выдает клиенту **502 Bad Gateway**.

### 🛠 Как это было решено (Паттерн Лечения)
- **Ошибка развертывания:** Мягкое переключение (`docker-compose up -d --no-deps app`) было выполнено ДО того, как изменения схемы базы данных были применены.
- **Официальный Шаг Решения:** В скрипт выкатки (`deploy-smmplan.sh`) необходимо внедрять строгое правило: КОМАНДА миграции или применения схемы `npx prisma db push / migrate deploy` **ОБЯЗАНА** запускаться либо в процессе сборки самого Docker-контейнера в виде `prestart` скрипта, либо вызываться оркестратором (отдельный transient container) **СТРОГО ДО** скачивания и перезапуска Production Node.js контейнера.

## 5. Требования к Оркестратору (Rollout Script)
Скрипт `scripts/deploy-smmplan.sh` должен использовать:
1. `docker compose pull` (скачка без остановки старой версии).
2. **ОБЯЗАТЕЛЬНО:** Выполнение `npx prisma migrate deploy` через временный контейнер на старой структуре (используя Expand-Contract).
3. Замедленный старт (Staggered boot): `docker compose up -d --no-deps app`.
4. Обязательный **Health Check** (polling `api/health` 5 раз). Если 502 — скрипт применяет команду Rollback и возвращает дамп БД.

## 6. Катастрофы CI/CD и Архитектуры (Историческая Память)
Строго запрещено допускать следующие сбои инфраструктуры, которые уже приводили к полному отказу (100% 502/429):

### I. Перехват Сборки (Web vs Webhook Bot)
**Суть бага:** Из-за того, что в `Dockerfile` стадия сборки Telegram-бота (`bot-runner`) была написана ниже основной стадии (`runner`), Docker по-умолчанию собирал и деплоил `bot-runner` в веб-контейнер `app:3000`. В результате Nginx стучался на вебсайт, а там был Express-сервер вебхуков, отдававший пустой `404 Not Found` с жестким заголовком `Content-Security-Policy: default-src 'none'`.
**Правило выживания:** 
Всегда явно прописывайте `target: runner` у веб-сервисов в `docker-compose.yml` и убеждайтесь, что стадия `runner` находится в САМОМ НИЗУ многостадийного `Dockerfile`.

### II. Смерть в Зачатке: Sentry DSN Crash
**Суть бага:** Файл `instrumentation.ts` инициализировал библиотеку `@sentry/nextjs` с плейсхолдером `https://placeholder-dsn@sentry.io/0`, если рабочий ключ отсутствовал. Новое ядро Sentry парсило этот URL, признавало его "Invalid Dsn", бросало `throw new Error()` и жестко убивало Node.js процесс в 0 миллисекунд после запуска. Healthcheck рушился, контейнер уходил в циклический рестарт `unhealthy`.
**Правило выживания:** 
**Никогда** не передавайте плейсхолдеры в `Sentry.init()`. Делайте строгую проверку: `if (process.env.SENTRY_DSN) { Sentry.init({...}) } else { // ignore }`.

### III. Фатальный Fail-Closed в Rate Limiter (Инцидент 429)
**Суть бага:** Rate Limiter (`@upstash/ratelimit`) в Middleware был спроектирован так, что если отсутствовали ключи `UPSTASH_REDIS_REST_URL` (например, при локальном деплое Docker Redis), он переходил в "Fail-Closed" режим и выдавал `429 Too Many Requests` АБСОЛЮТНО ВСЕМ посетителям из соображений безопасности.
**Правило выживания:** 
Rate Limiter в пограничных функциях должен иметь **Memory Fallback** (например, через V8 `new Map()`), чтобы в случае потери связи с Redis он продолжал защищать сервис локальными счетчиками, а не ложил всю бизнес-логику в паралич.

### IV. Десинхронизация Docker Volumes (404 на Статику)
**Суть бага:** При пересборке Next.js генерирует новые хэши для CSS/JS чанков (папка `.next/static`). Однако Nginx кэшировал статику через замороженный Docker Volume (`next_static`). В результате, новый `index.html` запрашивал свежие файлы, Nginx искал их в замороженном томе, не находил и выдавал `404 Not Found`.
**Правило выживания:** 
Вместо сложной синхронизации файлов с Nginx Volumes, используем сам Next.js для раздачи своей же статики. Если используется Cloudflare, он самостоятельно закэширует всю статику. Nginx **не должен** агрессивно перехватывать вызовы `/_next/static/` без необходимости.

### V. Катастрофа Фантомного Кода (Рассинхрон Локаль/Продакшн)
**Суть бага:** Разработчик (или Агент) вносил критические хотфиксы в исходный код (`instrumentation.ts`, `rate-limiter.ts`), но забывал пересобрать Production Docker-образ. В итоге локальная кодовая база идеальна, но сервер продолжал бесконечно падать со старыми ошибками (Sentry Dsn Crash, Fail-Closed), исполняя замороженный кэш старой сборки.
**Правило выживания:** 
Модификация серверной бизнес-логики (`.ts`, `next.config.mjs`) **мёртва**, пока не упакована в образ. Вы ОБЯЗАНЫ вызвать CI/CD-пайплайн (например, залив тегом `git tag vX.Y.Z && git push origin vX.Y.Z`) и дождаться выкатки нового образа на сервер! Фиксы "по месту" не работают в Docker-среде.
