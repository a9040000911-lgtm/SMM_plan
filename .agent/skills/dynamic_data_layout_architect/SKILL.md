# Dynamic Data Layout Architect

## Problem Context
Приложение Smmplan оперирует финансовыми показателями (B2B микро-транзакции), юзер-сгенерированным контентом (имена, отзывы) и динамическими каталогами провайдеров API. 
Жесткая верстка с использованием `truncate`, `line-clamp-1` или фиксированными ширинами (`w-32`) часто приводит к **потере критической информации** для пользователя.
*Например, обрезание суммы `123,45... ₽` в корзине разрушает доверие и приводит к ошибкам при оплате.*

## The Rules of Layout Flexibility

### 1. Zero Truncation for Financial Data (Prices, Balances, Totals)
**НИКОГДА** не используйте `truncate` или `w-[fixed]` для отображения финальных цен, балансов или количества.
- **Bad:** `<span className="text-xl truncate w-24">{price}</span>`
- **Good:** `<span className="text-xl whitespace-nowrap shrink-0 max-w-[50%] md:max-w-none">{price}</span>`
- Если контейнер ограничен, используйте адаптивный размер шрифта (Responsive Typography) `text-sm sm:text-base md:text-lg`.
- Для больших сумм разрешайте перенос валюты или "хвоста" на новую строку, но сами цифры держите вместе: `whitespace-nowrap`.

### 2. Flex-based "Safe Zones" for Sidebars and Footers
Для сайдбаров, админ-панелей и карточек с подвалами (Footer) избегайте наложения, используя паттерн **Flex Column with Locked Footer**:
- **Wrapper:** `flex flex-col h-full`
- **Scrollable Area:** `flex-1 overflow-y-auto min-h-0` (Обязательно `min-h-0` для корректного сжатия flex-блока!).
- **Safe Zone Padding:** В скроллируемой зоне всегда добавляйте `pb-16` или `pb-20`, чтобы последний элемент не перекрывался тенями или футером.
- **Locked Footer:** `shrink-0 min-h-[4rem]` - Футер не должен сжиматься, его минимальная высота жестко зафиксирована.

### 3. Graceful Multiline Expansion for Dynamic Text
(Описания услуг, отзывы, длинные заголовки)
- Не ограничивайте высоту контейнера описания: избегайте `h-16 overflow-hidden`.
- Если нужно ограничение по UI, используйте `line-clamp-3 md:line-clamp-5` для превью, и давайте кнопку разворачивания (Expand) для полного текста.
- При выводе чистых описаний провайдеров из базы, учитывайте многострочный формат:
  - Использовать `whitespace-pre-wrap` для корректного отображения `\n`.
  - Указывать `leading-relaxed` (или 1.6) для читаемости.
  - Допускается обнуление обтеканий `break-words`.

### 4. Interactive & Multiplicative Inputs (Smart Quantity / Gradients)
При создании кнопок или пресетов, которые умножают цену, они должны:
1. Вычисляться динамически на основе `minQty` и `step`. Нельзя хардкодить `[100, 500, 1000]`, так как у услуги может быть `step = 5` или `step = 0.1` (если это часы).
2. UI-кнопки должны использовать Flex-grid (`flex-1` или `grid-cols-X`) и растягиваться пропорционально, а не оборачиваться (wrap) некрасиво на мелких экранах.
3. Для очень больших чисел в кнопках (например, миллионы) использовать аббревиатуры `min >= 1000 ? {min/1000}K : min`, чтобы не разрывать UI.

## Diagnostics Checklist
При код-ревью или создании новых UI компонентов, задайте три вопроса:
1. Что произойдет с этим элементом, если число станет `99 999 999,99`? Оно обрежется? Если да - рефакторим.
2. Что произойдет, если пользователь - немец с именем "Wolfeschlegelsteinhausenbergerdorff"? Он сломает сайдбар? Если да - используем `text-ellipsis` и сжатие только для этого имени, но не для функциональных кнопок.
3. Достают ли элементы до самого низа при скролле (особенно на мобильных), или их перекрывает нижняя панель навигации (Bottom Nav TabBar)? Добавьте `pb-safe`.
