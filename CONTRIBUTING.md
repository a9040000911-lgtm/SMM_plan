# Руководство Контрибьютора (CONTRIBUTING) 🤝

Добро пожаловать в команду разработки **Smmplan**! 

Это руководство содержит стандарты и правила, которым подчиняется весь процесс разработки. Мы стремимся к максимальной стабильности (Zero-Error Policy) и высокому качеству продуктовой логики.

---

## 🧭 Методология Разработки: The Ralph Loop

Все агенты-разработчики и инженеры в проекте используют фреймворк **Ralph Loop v3**. Эта методология защищает проект от регрессий и слепых зон при разработке.

### 6 шагов стандарта Ralph Loop:
1. **Holistic Audit (Аудит логики, UX и Безопасности):** Перед написанием кода, задача анализируется через призму общей архитектуры (Влияет ли это на другие проекты? Есть ли риск SSRF/IDOR? Страдает ли UX?).
2. **Decomposition (Декомпозиция):** Масштабная задача разбивается на проверяемые истории (User Stories).
3. **Iterative Solution (Итеративное решение):** Код пишется атомарно: шаг за шагом.
4. **Task Verification (Проверка шагов):** Каждый шаг проверяется линтерами, тайп-чекерами и тестами перед тем, как идти дальше.
5. **Final Integrity Check (Проверка целостности проекта):** Обязательный запуск общепроектного сбора (build) и `eslint`.
6. **Persistence:** Фиксация результатов (запись в историю, создание PR).

## 🛡 Безопасность (Security Context)

Проект отвечает за высоконагруженные финансовые и провайдерские операции (заказы услуг). 

При написании кода обязательно:
- **Zod-валидация**: Строгая типизация и парсинг всех входных данных (API endpoints, Server Actions, Telegram-команды). Никаких `any`. 
- **Server Actions Guard**: В Next.js `Server Actions` запрещено доверять клиентскому стейту. Данные должны перепроверяться (в том числе права админа).
- **Изоляция проектов (Multi-Project Architecture)**: Если вы пишите код для Internal Services или админки, учитывайте контекст (`projectId`), чтобы клиенты не видели цены других пользователей.

## 📝 Требования к коду и Техстеку

- **Next.js 16 + React 19:** Используем современный App Router. Никаких классовых компонентов.
- **Tailwind CSS 4.0:** Использование `@theme` вместо старого `tailwind.config.js`. Не забывайте про семантику и принципы UI-архитектора при верстке.
- **ESLint 10 (Flat Config):** Код должен проходить проверку без предупреждений (`npm run lint`). Конфигурация находится в `eslint.config.mjs` (в виде массива).
- **TypeScript 5.7+:** Строгий режим (`strict: true`). Использование `unknown` вместо `any`.

## 🌿 Git Flow и правила управления ветками

1. Работа над новой фичей всегда ведется в ветках `feature/название-фичи` (например: `feature/loyalty-system`).
2. Баги правятся в ветках `bugfix/название-проблемы` или `hotfix/название` (если срочно в продакшен).
3. Именование коммитов согласно стандарту **Conventional Commits**:
   - `feat: добавил систему бейджей лояльности`
   - `fix: исправил обрыв API-клиента YooKassa`
   - `docs: обновил описание архитектуры в Diátaxis`
   - `refactor: перенос OrderProcessor на паттерн делегирования`

## 🚀 Проверка перед Pull Request

Перед созданием Пулл-Реквеста (PR) убедитесь, что вы выполнили локальные проверки (Final Integrity Check):

```bash
# Проверка типов
npm run build

# Линтинг по новым стандартам ESLint 10
npm run lint

# (Если есть тесты)
npm run test
```
