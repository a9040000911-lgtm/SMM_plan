# Логика определения и анализа ссылок (Link Analyzer)

## 🏗️ Как система понимает, какой тип ссылки нужен услуге?

Определение требуемого типа ссылки происходит на уровне базы данных и привязки к категории.

1.  **InternalService (Услуга)**:
    *   Поле `targetType`: Основной требуемый тип (например, `CHANNEL`, `POST`, `STORY`). По умолчанию — `POST`.
    *   Поле `allowedTargetTypes`: Массив строк, содержащий дополнительные допустимые типы. Если услуга универсальна (например, "Просмотры на весь контент"), здесь могут быть перечислены несколько типов.
2.  **ServiceCategory (Категория)**:
    *   Если у конкретной услуги тип не задан или нужно определить значение по умолчанию при импорте, система смотрит на `targetType` категории.

### Основные системные типы (TargetType):
*   `CHANNEL` — Каналы, группы, паблики, сообщества.
*   `POST` — Обычные публикации, посты в ленте, твиты.
*   `VIDEO` — Видеоролики (YouTube, Reels, Clips).
*   `STORY` — Сторис и моменты.
*   `COMMENT` — Ссылки на конкретные комментарии.
*   `CUSTOM` / `EXTERNAL` — Специфические ссылки (боты, прокси, внешние сайты).

## 🔍 Как работает Анализатор Ссылок (Link Analyzer)?

Процесс проверки ссылки при создании заказа (`Order`) состоит из трех этапов:

### 1. Нормализация (Normalization)
Система очищает ссылку от "мусора":
*   Добавляет `https://`, если юзер ввел просто домен.
*   Удаляет трекинговые и рекламные параметры: `utm_*`, `igsh`, `ref`, `source`, `list`.
*   Удаляет висящие слэши на конце.
*   Приводит всё к нижнему регистру (для доменов).

### 2. Распознавание платформы и объекта (Parsing)
Система использует набор парсеров для каждой платформы (`TelegramParser`, `VkParser`, `InstagramParser` и т.д.):
1.  **Поиск парсера**: По домену (например, `t.me`, `vk.com`, `instagram.com`) выбирается нужный алгоритм.
2.  **Анализ пути (Path)**:
    *   `t.me/c/123/456` → `TG_POST` (приватный пост).
    *   `t.me/smmplan/123` → `TG_POST` (публичный пост).
    *   `t.me/smmplan` → `TG_CHANNEL`.
    *   `vk.com/wall-123_456` → `VK_WALL` (пост на стене).

### 3. Маппинг и Валидация
Полученный технический объект (например, `IG_REEL`) сопоставляется с системным типом (`VIDEO`).
Затем выполняется финальная проверка:
*   **Совпадение платформ**: Если услуга для Telegram, а ссылка от Instagram — заказ будет отклонен.
*   **Совпадение типов**: Если услуга требует `POST`, а прислана ссылка на `CHANNEL` — система выдаст ошибку.

## ❓ Что будет, если ссылка не распознана?

Если анализатор не может однозначно определить тип ссылки, поведение зависит от контекста:

1.  **Неизвестный домен**:
    *   Если ссылка не принадлежит ни одной из известных соцсетей, она получает тип `OTHER` и объект `WEB_SITE`.
    *   Это позволяет пропускать ссылки для услуг типа "Трафик на сайт".
2.  **Неверный формат**:
    *   Если это вообще не URL и не понятный handle (например, `просто текст`), возвращается ошибка: **"Неверный формат ссылки"**.
3.  **Fallback (Запасной вариант)**:
    *   Если домен распознан (например, `vk.com`), но путь странный, система по умолчанию ставит тип `CHANNEL` (как самый универсальный для подписок).
4.  **Ошибка в интерфейсе**:
    *   Пользователь увидит понятное сообщение, например: *"Для этой услуги нужна ссылка на пост/публикацию, а вы прислали ссылку на канал/сообщество"*.
