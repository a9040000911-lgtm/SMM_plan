# План лицензирования и защиты проекта Smmplan (2026)

## Выбранная стратегия безопасности
С учетом специфики базы кода (Next.js 16+, Multi-Tenant Архитектура) выбрана стратегия **SaaS + White-Label**. 
Исходный код остается на наших серверах (не передается клиенту в виде коробочного Docker-компонента), а клиенты получают изолированные пространства (`projectId`) с привязкой собственных доменов (CNAME). Это гарантирует невозможность прямого вырезания лицензии из кода (Cracking).

## Задачи на внедрение (Беклог)

### 1. Edge-защита от парсинга БД (Анти-скрейпинг API)
- Внедрить **Rate Limiting** в `src/middleware.ts` поверх Edge/Worker среды (рекомендуется `@upstash/ratelimit`).
- Настроить строгий **Cloudflare WAF**: включить Bot Fight Mode и проверку DCoM (data center) IP-адресов. Легитимный трафик B2B чаще идет с мобильных или домашних провайдеров, а парсеры – с AWS/DigitalOcean.

### 2. Защита от "складчин" (Shared Accounts Banning)
- Добавить механизм **Device Fingerprinting** на момент входа (NextAuth / JWT).
- Вшивать в сессионный токен хэш-отпечаток (UserAgent + подсеть + языковые настройки). 
- Если токен одного пользователя фиксируется одновременно (overlap-сессия) с кардинально разных Fingerprint-хэшей, старые сессии инвалидируются. Реализовать правило "Одна активная сессия на один аккаунт".

### 3. Харденинг Server Actions и Бизнес-логики
- Все запросы, приходящие в `src/app/admin/security/actions.ts`, `orders/actions.ts` и аналогичные, должны проходить через строгий фильтр `src/utils/service-sanitizer.ts`.
- Внедрить валидацию CSRF-токенов при вызове критических мутаций (изменение цен, балансов, удаление провайдеров).

### 4. Юридическая защита и Авторское право
- **Регистрация кода:** Задепонировать критические узлы модулей (финансовое ядро, геймификацию, логику API-провайдеров) в Реестре программ для ЭВМ (Роспатент). Свидетельство — лучший рычаг давления на абузоустойчивых хостеров, если кто-то попытается "скопировать" Smmplan.
- **Оферта B2B:** Добавить в Terms of Service чекбокс с прямой материальной ответственностью (неустойкой) клиента за автоматизированный парсинг каталога услуг и подделку API-запросов.
