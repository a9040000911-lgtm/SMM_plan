# 🛠 Практическая инструкция: Настройка и управление SMM-Провайдерами

**Smmplan** работает как интеллектуальный маршрутизатор: мы получаем заказ от клиента и автоматически отправляем его внешним SMM-провайдерам (например, оптовым поставщикам услуг). В этой инструкции описан алгоритм добавления и настройки связи с провайдерами через API.

## Шаг 1: Архитектура делегирования (ProviderService)

Для взаимодействия с внешними вендорами (в т.ч. SMM Panel API) мы используем класс `ProviderService`. Он основан на паттерне "Адаптер/Делегат", который позволяет легко подключать новых провайдеров, не ломая основную логику оформления заказа (OrderProcessor).

Для каждого вендора создается свой драйвер, реализующий общий контракт:
- Метод `createOrder(data)`
- Метод `checkStatus(providerOrderId)`
- Метод `getBalance()`

## Шаг 2: Получение и настройка API ключей

Для того чтобы платформа могла заказывать услуги, необходимо прописать ключи доступа в конфигурации.

1. Откройте панель администратора Smmplan: `http://localhost:3000/admin`.
2. Перейдите в раздел **Providers (Провайдеры)**.
3. Добавьте нового провайдера (например, *TestProvider*).
4. Укажите `API Endpoint` (например, `https://testprovider.com/api/v2`).
5. Укажите `API Key` (ваш персональный токен, выданный провайдером).

> [!WARNING]
> Ключи API провайдеров хранятся в БД в зашифрованном виде. Не логируйте конфигурации `ProviderService` в системную консоль в чистом виде (`console.log(config)`), чтобы не произошла утечка ключей!

## Шаг 3: Настройка Отказоустойчивости (Failover Logic)

Smmplan спроектирован с упором на стабильность. Если первый провайдер недоступен (502 Bad Gateway) или выдает ошибку API (Not enough balance), система не должна отменять заказ клиента. Вместо этого включается режим **Failover**.

### Включение Failover Mode

Чтобы заказ автоматически "перекидывался" на запасного провайдера:

1. В админ-панели перейдите в **Service Catalog (Услуги)**.
2. При создании/редактировании услуги (Например: *YouTube Likes*), добавьте нескольких "Внутренних" поставщиков (Internal Providers).
3. Установите приоритеты: `Priority 1 (Main)`, `Priority 2 (Backup)`.

### Итог (User Flow)
Когда клиент покупает услугу, `OrderProcessor` попытается списать средства у провайдера `Priority 1`. Если метод `createOrder()` выбросит исключение (exception), включается `FailoverService` и автоматически перезапускает транзакцию с провайдером `Priority 2`. Администратор получит уведомление в Telegram (канал "Ошибки").

## Шаг 4: Безопасность и SSRF

Мы строго валидируем все исходящие запросы к провайдерам. Перед запросом убедитесь, что URL-адрес провайдера проходит через жесткий валидатор `URL_REGEX`. 

> [!CAUTION]
> Интеграция сторонних API без должной проверки доменов (отслеживание, куда реально идет axios/fetch запрос) может создать SSRF-уязвимость (Server-Side Request Forgery). Доверяйте только строго определенным Endpoint'ам.
