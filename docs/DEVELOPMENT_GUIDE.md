# Руководство по разработке (Development Guide)

## 🏗 Инфраструктура (Infrastructure)
Для корректной работы приложения требуются следующие сервисы:
1. **PostgreSQL**: Основная база данных.
2. **Redis**: Используется для:
   - **Rate Limiting**: Ограничение частоты запросов к API.
   - **Queues (BullMQ)**: Фоновая обработка заказов и синхронизация статусов.
   - **2FA**: Хранение временных кодов авторизации.

### Быстрый запуск через Docker
```bash
docker-compose up -d
```
Это запустит PostgreSQL и Redis с необходимыми параметрами.

## 📌 Основные принципы
1. **Security First**: Всегда проверять права доступа (adminId) и защищать транзакции от Double-click.
2. **Precision**: Использование `Decimal.js` для всех финансовых операций. Никаких `Float` для денег.
3. **White-label**: Клиент не должен видеть названий внешних провайдеров или технических ошибок API.
4. **Transparency**: Любое изменение баланса или статуса должно логироваться (`AdminLog`).

## 🛠 Стандарты кодирования

### 1. Именование (Naming)
- **Файлы**: `kebab-case.ts` (e.g., `order-processor.service.ts`).
- **Функции/Переменные**: `camelCase`.
- **Классы/Интерфейсы**: `PascalCase`.
- **БД (Prisma)**: Модели в `PascalCase`, поля в `camelCase`.

### 2. Работа с БД (Prisma)
- Всегда использовать `$transaction` при изменении баланса и создании заказа.
- Включать связанные сущности (`include`) только когда это необходимо для производительности.
- Использовать атомарные обновления: `{ balance: { increment: 100 } }` вместо `balance + 100`.

### 3. Обработка ошибок (Error Handling)
- Использовать `try-catch` блоки в фоновых задачах и экшенах бота.
- Фатальные ошибки отправлять админу через `bot.catch`.
- Для пользователя выводить `user-friendly` сообщения на русском языке.

### 4. Документация изменений
- После реализации фичи обновить `PROGRESS.md`.
- Если изменилась схема БД — описать это в соответствующем модуле `docs/*.md`.

## 📦 Работа с провайдерами
- Любой новый провайдер должен наследоваться от `IProvider` в `src/services/providers/base-provider.ts`.
- Реализация должна обрабатывать специфичные ошибки API и приводить их к стандарту `OrderResult`.

## 🛡 Безопасность
- **Double-click Protection**: Использовать `processingUsers` Set в боте или Redis-блокировки в API.
- **Input Validation**: Валидация ссылок через `analyzer.ts` и `normalizer.ts`.
- **Margin Guard**: Всегда проверять рентабельность заказа перед его финальным оформлением.
