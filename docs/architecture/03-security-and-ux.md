# 🏛 Архитектура: Безопасность и Neuro-UX

Кросс-функциональные требования системы, которые пронизывают **Smmplan** от интерфейса до базы данных.

## 1. Концепция Neuro-UX

Помимо чистой функциональности, в Smmplan интегрирована философия высококонверсионного интерфейса. Мы не делаем 'просто кнопки'. 
Наши чек-ауты спроектированы на основе нейромаркетинга:
- **Smart Presets:** Услуги по умолчанию подсвечиваются, снижая когнитивную нагрузку при выборе.
- **Trust Signals (Сигналы доверия):** Плавные индикаторы загрузки, четкие сообщения об ошибках.
- **Micro-Animations:** Психологическое подкрепление действия клиента (например, анимация "Заказ принят"). 
- **Scroll Minimization:** Все самые важные элементы находятся в первой области видимости. Для заказа не нужно скроллить 3 страницы.

*См. навык `scroll_optimization` в папке `.agents/`.*

## 2. Модель Угроз и Защита (Threat Modeling)

Так как Smmplan управляет финансовыми балансами B2C и API-токенами B2B, мы внедрили строгие протоколы:

### А. Защита от SSRF (Server-Side Request Forgery)
Когда администратор вводит URL Provider API, система может стать жертвой SSRF-атаки, если URL указывает на внутреннюю сеть (`localhost`, `127.0.0.1`, `169.254.169.254`). 
- **Решение:** Мы валидируем IP адреса перед тем, как `ProviderService` сделает `axios`/`fetch` вызов (или используем жесткие белые списки).

### Б. IDOR и Изоляция Проекта
(Insecure Direct Object Reference)
- В `GET /api/orders/[id]` сервер обязательно сверяет `order.userId === session.userId` (или `order.projectId`). 
- Ни один клиент не может увидеть статистику или тикет поддержки (HappinessDesk) другого клиента, перебрав `id` в URL.

### В. Server Actions
Next.js `Server Actions` — это скрытые POST-запросы.
- **Уязвимость:** Злоумышленник может вызвать `deleteTicket(ticketId)` просто скопировав функцию, даже если на фронтенде этой кнопки для него нет.
- **Защита (Guards):** Внутри каждой Server Action функции мы вручную проверяем `Cookies / Role` перед SQL запросом:
  ```typescript
  export async function deleteTicket(id: string) {
    const session = await getSession();
    if (session.role !== "ADMIN") throw new Error("Forbidden");
    // ... delete logic
  }
  ```

Мы живем по правилу: **"Не доверяй клиенту" (Zero Trust Principle)**. Любые входящие данные проходят Zod схему. Любая Server Action проверяет аутентификацию.
