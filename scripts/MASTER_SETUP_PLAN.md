# 🏁 Генеральный план запуска Smmplan (Owner Edition)

Этот документ описывает точную последовательность действий для вас, как владельца проекта.

## 📍 Текущий статус
Мы уже провели глубокую оптимизацию кода, ИИ-аудит услуг и подготовили все конфигурации. Весь код закоммичен в Git.

---

## 🗺️ Дорожная карта (Шаг за шагом)

### Шаг 1: Локальная проверка (Ваш компьютер)
**Цель:** Убедиться, что всё работает идеально перед тем, как показывать это миру.
1.  **Запуск**: Используйте гибридный режим (Docker для БД + Node.js для кода).
    - Инструкция: [LOCAL_DEVELOPMENT.md](file:///d:/Smmplan/LOCAL_DEVELOPMENT.md)
2.  **Тесты**: Создайте тестовый заказ в боте и посмотрите, как он отображается в админке на `localhost:3000`.

### Шаг 2: Выход в "Продакшн" (Сервер Ubuntu)
**Цель:** Запустить проект на реальном железе.
1.  **Перенос**: Соберите `.tar` образы на своем ПК и перекиньте их на сервер через WinSCP.
    - Инструкция: [DEPLOY_2.md](file:///d:/Smmplan/DEPLOY_2.md).
2.  **Запуск**: Выполните `docker compose up`.
3.  **Безопасность**: Убедитесь, что порты базы данных закрыты (я это уже настроил в конфигах).

### Шаг 3: Настройка инфраструктуры GitLab
**Цель:** Создать среду, в которой команда сможет пушить код.
1.  **Установка**: Разверните GitLab прямо на вашем Ubuntu сервере.
    - Конфиг: [docker-compose.gitlab.yml](file:///d:/Smmplan/docker-compose.gitlab.yml)
    - Инструкция: [GITLAB_SETUP.md](file:///d:/Smmplan/docs/infrastructure/GITLAB_SETUP.md)
2.  **Загрузка кода**: Сделайте первый `git push` вашего готового проекта в ваш новый GitLab.

### Шаг 4: Подключение команды (Dev Team)
**Цель:** Передать проект разработчикам и начать работу по Agile.
1.  **Презентация**: Покажите им [TEAM_PRESENTATION.md](file:///d:/Smmplan/docs/collaboration/TEAM_PRESENTATION.md).
2.  **Обучение**: Выдайте им [HANDBOOK.md](file:///d:/Smmplan/docs/collaboration/HANDBOOK.md).
3.  **Первая задача**: Попробуйте вместе пройти цикл: создание Issue -> Ветка -> Merge Request.

---

## ⚡ Причинно-следственная связь (Почему так?)
1. **Сначала Локально** => Чтобы не тратить время на "битые" сборки на сервере.
2. **Потом Деплой** => Чтобы проект уже начал приносить пользу, пока настраивается GitLab.
3. **GitLab в конце** => Это сложный инструмент. Его лучше настраивать, когда основной продукт уже стабильно работает.
