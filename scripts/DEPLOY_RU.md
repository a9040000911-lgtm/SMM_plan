# Инструкция по миграции Smmplan на Linux (Docker)

Эта инструкция поможет вам развернуть проект на сервере с ОС Linux, решив проблемы с отсутствующими стилями и стабильностью работы.

## 1. Что изменилось (почему теперь будет работать)
*   **Прямая отдача статики**: Nginx теперь сам отдает CSS, JS и картинки через общие тома (Volumes), не нагружая Node.js. Это решает проблему "неподгружающихся стилей".
*   **Build Args**: Переменные `NEXT_PUBLIC_URL` теперь передаются на этапе сборки образа, что критично для работы фронтенда.
*   **Автоматизация**: Весь процесс сборки и настройки БД теперь упакован в один скрипт.

## 2. Подготовка сервера
Убедитесь, что на сервере установлены:
*   **Docker** (v20+)
*   **Docker Compose** (v2+)
*   **Git**

## 3. Шаги развертывания

### Шаг 1: Копирование файлов
Перенесите файлы проекта на сервер. **ВАЖНО!** Не копируйте следующие папки и файлы (они замедлят загрузку и не нужны для работы):
*   `node_modules/` — всё установится внутри Docker.
*   `.next/` — проект соберется заново в контейнере.
*   `.git/` — история разработки не нужна на сервере.
*   `.env` — **не копируйте свой локальный файл!** Создайте новый на сервере.
*   `backups/`, `*.sql`, `*.tar` — старые бэкапы БД.
*   `logs/`, `*.log` — файлы логов.
*   `docs/`, `.agent`, `.GEMINI` — документация и файлы ИИ-агентов.
*   `playwright-report/`, `coverage/` — отчеты по тестам.
*   `_trash/`, `tmp/` — временные файлы.
### Шаг 2: Настройка окружения
Если у вас еще **нет домена**, используйте IP-адрес сервера (например, `http://95.213.x.x`).

Создайте файл `.env` в корне проекта на основе `.env.example` и заполните его:
```bash
cp .env.example .env
nano .env # Отредактируйте переменные
```
**Важные поля для IP-доступа:**
*   `NEXT_PUBLIC_URL=http://<IP_ВЕРВЕРА>` (без слеша в конце).
*   `NEXTAUTH_URL=http://<IP_ВЕРВЕРА>`.
*   `WEBAPP_URL=http://<IP_ВЕРВЕРА>`.

### Шаг 3: Запуск миграции
Просто запустите созданный скрипт автоматизации. Он сделает всё за вас:
```bash
chmod +x scripts/migrate-server.sh
./scripts/migrate-server.sh
```

## 4. Что делает скрипт миграции?
1.  **Сборка**: Собирает Docker-образы, прокидывая ваш домен внутрь для корректной работы Next.js.
2.  **Запуск**: Поднимает 5 контейнеров: `nginx`, `app`, `bot`, `db`, `redis`.
3.  **База данных**: Автоматически применяет схему Prisma (`db push`) и патчи.

## 5. Проверка работы
*   **Статус контейнеров**: `docker compose -f docker-compose.prod.yml ps` (все должны быть Up).
*   **Логи приложения**: `docker logs -f smmplan-app`.
*   **Логи бота**: `docker logs -f smmplan-bot`.

## 7. Работа в команде (GitLab на этом же сервере)
Так как GitLab уже развернут на вашем сервере, флоу будет максимально быстрым:

1.  **Доступ к GitLab**: Заходите в GitLab по `http://<IP_СЕРВЕРА>:8080`.
2.  **Регистрация Runner**: 
    Чтобы авто-деплой заработал, нужно связать GitLab с Docker на сервере:
    ```bash
    # Запуск внутри сервера
    docker exec -it gitlab-runner gitlab-runner register
    ```
    *   **URL**: `http://gitlab:80` (внутренняя сеть docker)
    *   **Token**: возьмите в Settings -> CI/CD -> Runners в вашем проекте.
    *   **Description**: `smmplan-shell`
    *   **Tags**: `smmplan-shell`
    *   **Executor**: `shell`
3.  **Авто-деплой**: Теперь любой `git push origin main` будет запускать мой скрипт `migrate-server.sh` и обновлять сайт за 1-2 минуты.
